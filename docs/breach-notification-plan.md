# データ侵害（ブリーチ）通知計画

作成日: 2026-04-07  
準拠法: GDPR Art. 33 / 34、個人情報保護法（日本）令和4年改正

---

## 前提

- **運営体制**: 個人運営（専任セキュリティチームなし）
- **主要インフラ**: Google Cloud Run / Cloud SQL / Cloud Logging / Cloudflare R2
- **連絡先**: nibankougen@gmail.com
- **監査ログ**: `admin_audit_logs` → Cloud Logging にリアルタイム転送済み
- **バックアップ**: Cloud SQL PITR 35 日間有効、RPO 1 時間

---

## 1. 「ブリーチ」の定義

本計画では以下を「データ侵害（ブリーチ）」として扱います。

| 種別 | 具体例 |
|---|---|
| 機密性の侵害 | DB への不正アクセス・認証情報の漏洩・アカウント乗っ取りによる個人データ閲覧 |
| 完全性の侵害 | データの不正改ざん（ユーザー情報・購入履歴等） |
| 可用性の侵害 | ランサムウェア・DB 削除によるデータへのアクセス不能 |

---

## 2. リスク分類（通知要否の判断基準）

すべてのインシデントはリスク分類を行い、通知義務を判定します。

### レベル 1（低リスク）：記録のみ、監督機関への報告不要

以下の**すべて**を満たす場合：
- 漏洩したデータが暗号化済みで復号不能（TLS 転送中の傍受など）
- 影響を受けたユーザーが特定できない
- 個人情報への実質的なアクセスが発生していない

例: TLS ダウングレード試行（失敗）、ブルートフォース試行（ブロック済み）

### レベル 2（中リスク）：監督機関へ報告必要

- メールアドレス・表示名・IP アドレス等の個人情報が漏洩した可能性がある
- 影響ユーザー数が少数（概ね 1,000 人未満）かつセンシティブデータではない

例: API 脆弱性による一部ユーザー情報への不正アクセス

### レベル 3（高リスク）：監督機関への報告 + ユーザーへの個別通知

以下のいずれかに該当する場合：
- 大規模漏洩（1,000 人以上のユーザーデータ）
- 認証情報（トークン・ソーシャルプロバイダー ID）の漏洩
- 購入履歴・コイン残高の改ざん
- 未成年者（`young_teen`）のデータへのアクセス
- DB への直接侵入・テーブル全件取得

## 3. 検知方法

現在のインフラで利用できる検知手段は以下のとおりです。

| 検知手段 | 仕組み | 対象 |
|---|---|---|
| **Cloud Monitoring アラート** | Cloud SQL / Cloud Run の異常メトリクス（CPU スパイク・エラーレート急増） | インフラ異常 |
| **Cloud Logging クエリ** | `admin_audit_logs` の異常操作パターン（深夜の大量 DELETE 等） | 管理者操作の異常 |
| **Go API エラーログ** | 認証失敗・権限エラーの急増 | 不正アクセス試行 |
| **ユーザーからの報告** | アプリ内通知・メール | アカウント乗っ取り等 |
| **Cloud Monitoring アップタイムチェック** | API 無応答の検知（既存設定あり） | サービス可用性 |

## 4. 対応タイムライン（72 時間以内）

```
T+0h   検知（アラート受信 or ユーザー報告 or 自己発見）
  ↓
T+1h   初期評価
         - 何が漏洩・改ざんされたか
         - 影響ユーザー数の概算
         - 現在進行中か（封じ込め優先）or 完了済みか
  ↓
T+2h   封じ込め措置
         - 侵害を受けたアカウント・トークンの無効化
           （active_users.token_revision インクリメントで全セッション強制ログアウト）
         - 必要に応じて Cloud Run サービス停止
         - Cloud SQL バックアップの保全（PITR スナップショット確保）
  ↓
T+4h   リスク分類確定（レベル 1 / 2 / 3）
  ↓
T+24h  【レベル 2 以上】監督機関への速報提出（詳細不明の項目は「調査中」で可）
  ↓
T+48h  影響範囲の詳細調査完了
         - Cloud Logging の監査ログで操作履歴を確認
         - 影響ユーザーリストの確定
  ↓
T+72h  【GDPR Art. 33 期限】監督機関への正式報告
       【レベル 3】影響ユーザーへの個別通知
  ↓
T+後日 インシデントレポートの作成・再発防止策の実施
```

## 5. 監督機関への報告

### 【選択肢 A】日本の個人情報保護委員会（PPC）のみ

**推奨条件**: 現時点で日本ユーザーのみを対象とするサービスである場合

- 報告先: 個人情報保護委員会（https://www.ppc.go.jp）
- 報告手段: PPC オンライン報告フォーム（https://www.ppc.go.jp/personal/anzen/report/）
- 日本法での報告義務発生条件（令和4年改正）:
  - 不正アクセスによる漏洩（件数不問）
  - 機微情報（センシティブデータ）の漏洩
  - 1,000 件超の漏洩
- 報告期限: 「速やかに（概ね3〜5日以内）」に速報 → 30 日以内に確報

### 【選択肢 B】PPC + EU 各国 DPA（Data Protection Authority）

**推奨条件**: EU/EEA 居住ユーザーが存在するまたは今後対象とする場合

- GDPR Art. 33 の72時間ルールが適用される
- 報告先: 影響を受けたユーザーの常居所地の EU 加盟国の DPA
  - 複数国にまたがる場合は「主管 DPA（Lead SA）」を定める必要があるが、個人運営でかつ EU に拠点がない場合は、最初に被害が出た国の DPA へ報告するのが現実的
  - EU DPA 一覧: https://edpb.europa.eu/about-edpb/about-edpb/members_en
- 報告期限: **72 時間以内**（日本法より厳格）

**現時点での推奨**: **選択肢 A**（日本向けサービスとしてローンチ）。EU 展開時に選択肢 B へ切り替え。ただし GDPR の72時間ルールをベースに運用しておくと、将来の移行コストが低い。

## 6. ユーザーへの通知（レベル 3 のみ）

GDPR Art. 34 が要求する「高リスクのブリーチ」の場合、影響ユーザーへ個別通知します。

### 通知手段の優先順位

| 手段 | 利用条件 | 実装状況 |
|---|---|---|
| **1. アプリ起動時強制モーダル** | GDPR Art. 34「直接通知」要件を満たす正式手段。次回起動時に必ず表示される | **実装済み**（`securityNotice` フィールド・`screens-and-modes.md` セクション 1.5.7） |
| **2. プッシュ通知**（APNs / FCM） | モーダルより先に届く補助手段。ユーザーが無効化している可能性あり | 実装済み |
| **3. アプリ内通知**（`notifications` テーブル） | プッシュ通知の補助。アプリを開けるユーザー向け | 実装済み |
| **4. メール** | メールアドレスが NULL 化されていない場合 | 送信機能の実装要確認 |

### 通知内容（必須記載事項：GDPR Art. 34）

```
件名: 【重要】お客様のデータに関するお知らせ

本文に含めること:
1. 侵害の内容（何が発生したか）
2. 漏洩した可能性のある情報の種類（メールアドレス / 表示名 / 購入履歴 等）
3. 講じた対処（セッション無効化・パスワードリセット等）
4. ユーザーが取るべき行動（注意事項）
5. お問い合わせ先: nibankougen@gmail.com
```

## 7. 監督機関への報告書テンプレート

GDPR Art. 33(3) が要求する記載事項：

```
1. 侵害の性質
   - 侵害の種別（機密性 / 完全性 / 可用性）
   - 影響を受けた個人データのカテゴリ
   - 影響を受けたユーザーの概算件数

2. データ保護担当者（DPO）の連絡先
   nibankougen@gmail.com

3. 侵害の可能性のある影響・結果の説明

4. 講じた / 講じる予定の対処措置
   - 封じ込め（トークン無効化・サービス停止等）
   - 原因除去（脆弱性修正）
   - 再発防止策

5. 報告時点で不明な情報（「調査中」と記載して後日補完可）
```

## 8. インシデント記録（報告の有無にかかわらず必須）

GDPR Art. 33(5) により、すべてのブリーチを記録する義務があります。
報告不要なレベル 1 のインシデントも含めて記録します。

記録先: `docs/incident-log.md`（非公開）または管理者専用の DB テーブル

| 記録項目 | 内容 |
|---|---|
| 発生日時 | 検知した日時 |
| 侵害の概要 | 何が起きたか |
| リスクレベル | 1 / 2 / 3 |
| 影響範囲 | 影響ユーザー数・データ種別 |
| 対処内容 | 封じ込め・修正・復旧の措置 |
| 監督機関への報告有無 | 報告日・報告先 |
| ユーザー通知有無 | 通知日・手段 |
| 再発防止策 | 実施内容・完了日 |

## 9. 検知強化（実装すべき設定）

詳細仕様: `docs/infra-abstract.md` セクション 10

**通知先:** メール（nibankougen@gmail.com）+ Discord（運営者専用サーバー）  
Discord 統合: Cloud Monitoring → Pub/Sub → Cloud Functions（`discord-notifier`）→ Discord Webhook

### レイヤー 1: Cloud Monitoring メトリクスアラート（コンソール設定のみ）

- [ ] `high-5xx-error-rate`: Cloud Run 5xx エラーレート 5% 超（5 分間）
- [ ] `high-sql-connections`: Cloud SQL 接続数 80% 超（5 分間）
- [ ] `high-sql-cpu`: Cloud SQL CPU 90% 超（5 分間）
- [ ] `api-uptime-check`: 既存アップタイムチェックを Discord にも通知するよう更新

### レイヤー 2: Cloud Logging ログベースアラート（コンソール設定のみ）

- [ ] `mass-admin-delete`: 1 時間以内に 100 件超の `delete_*` アクション
- [ ] `after-hours-admin-op`: 23:00〜7:00 JST に管理者操作が 1 件でも発生

### レイヤー 3: Go API 構造化ログ（コード実装 + Cloud Logging アラート）

- [ ] 認証ミドルウェア: レートリミット発動時に `brute_force_attempt` ログを出力
- [ ] トークン無効化処理: `mass_token_revocation` ログを出力（1時間100ユーザー超でアラート）
- [ ] レートリミットミドルウェア: `high_api_rate` ログを出力（同一ユーザー500req/分超）
- [ ] 上記 3 種を Cloud Logging ログベースアラートに設定
- [ ] 上記 3 種を `system_alerts` テーブルにも挿入（管理画面 8.9 に表示）

### 共通

- [ ] Pub/Sub トピック `lowpolyworld-security-alerts` を作成する
- [ ] `discord-notifier` Cloud Functions を実装・デプロイする
- [ ] Discord Webhook URL を Secret Manager に登録する
- [ ] 全アラートポリシーに Pub/Sub トピックを Notification Channel として追加する
