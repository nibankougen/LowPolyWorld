# 画面・モード仕様書

## 1. 基本設計方針

- **画面向き**: 縦画面（Portrait）固定で設計
- **ナビゲーション**: 画面下部のナビゲーションバー（5タブ）でモード切り替えを行う（セクション 1.6）
- **主なタブ**: ワールド / アバター管理 / ワールド管理 / ショップ / 設定
- **セーフエリア**: 原則としてすべてのボタン・テキスト・インタラクティブ要素は **Safe Area 内**に配置する。ノッチ・ダイナミックアイランド（iOS）やカメラパンチホール・ジェスチャーナビゲーションバー（Android）と重なる位置への配置は禁止する。Unity UI では `Screen.safeArea` を参照して UI ルートの RectTransform を調整する（SafeAreaFitter コンポーネントを全キャンバスに適用）。

### 1.1 共通 UI 配置ルール

#### 右上アイコンエリア（通知ベル + 自分のアイコン）

画面右上（Safe Area 内）に以下の2アイコンを横並びで常時表示する。左側が通知ベルアイコン、右側が自分のアイコン。

**通知ベルアイコン（左）**

- **非表示にする場面**:
  - 各種編集モード（ワールド編集画面・アバター編集画面・オブジェクト編集画面・テクスチャペイント画面）
  - ワールドモードの撮影モード中（セクション 2.7）

**自分のアイコン（右）**

- 自分の現在使用中アバターのサムネイルを円形で表示する
- **非表示にする場面**:
  - ワールドモード（ルーム内）
  - 各種編集モード（ワールド編集画面・アバター編集画面・オブジェクト編集画面・テクスチャペイント画面）
- **タップ動作**: アカウント表示モードへ遷移（セクション 22）

#### 共通モーダル UI ルール

アプリ内に表示されるすべてのモーダルに適用する共通規則。

- モーダル内のボタンは**縦に並べる**
- モーダル右上に **×ボタン** を配置し、タップでモーダルを閉じる
- **モーダル範囲外をタップ**した場合もモーダルを閉じる
- アクション完了時（通報・非表示・保存等）は画面上部に**フラッシュメッセージ**（トースト）で結果を表示する

#### 戻る（＜）ボタン

- **配置**: 画面左上（Safe Area 内）に表示する
- **表示する場面**: 前の画面へ戻る遷移が存在する画面すべて（ワールドモード内を除く）
  - ワールドモード（ルーム内）では戻るボタンを表示しない（退室はメニューから行う）
- **編集画面からの戻る動作**: ワールド編集・アバター編集・オブジェクト編集・テクスチャペイントなど編集中の画面から戻るボタンをタップした場合、以下の確認モーダルを表示する:

  > **「編集を終了しますか？」**
  > 保存していない変更は失われます。
  > ［保存して終了］　［破棄して終了］　［キャンセル］

  - 「保存して終了」: 編集内容を保存してから前の画面へ戻る
  - 「破棄して終了」: 変更を破棄して前の画面へ戻る
  - 「キャンセル」: モーダルを閉じ、編集を続ける（画面遷移しない）
  - 未編集状態（変更なし）の場合はモーダルを表示せず即座に戻る

## 1.5 タイトル画面（ログイン・アカウント作成）

タイトル画面はホーム画面と同一のシーンで構成する。アプリ起動時に最初に表示され、API 互換性チェックとログイン処理を担当する。ナビゲーションバーはタイトル画面では非表示とする。

### 1.5.1 API 互換性チェック

アプリ起動後、ログイン処理前に自動で API バージョンの互換性を確認する。

**API バージョンの方針:**

- API バージョンは整数値で管理する
- **バージョンが上がる条件**: 後方互換性がなくなる変更（既存エンドポイントの破壊的変更・レスポンス構造の変更等）
- **バージョンが上がらない条件**: 後方互換性を保った変更（新規エンドポイントの追加・任意パラメータの追加等）
- アプリのバージョンアップ時: 対応 API バージョンを最新の API バージョンに更新する
- 移行期間: 最低互換 API バージョンは最新バージョンとは限らない（旧バージョンのクライアントを一定期間サポート可能）

**互換性チェックフロー:**

1. `GET /api/version` でサーバーの最低互換 API バージョンを取得する
2. **最低互換バージョン > クライアントの対応バージョン** の場合 → アップデート促進モーダルを表示してログインをブロック
3. それ以外（互換あり）→ ログインフローへ進む

**アップデート促進モーダル:**

- メッセージ: 「このバージョンのアプリはサポートが終了しました。最新バージョンにアップデートしてください。」
- ボタン（OS ごとに出し分け）:
  - iOS: 「App Store でアップデート」→ App Store のアプリページへ遷移
  - Android: 「Google Play でアップデート」→ Google Play のアプリページへ遷移
- × ボタンなし・モーダル外タップ無効（ログイン操作に進めない）

### 1.5.2 ログインフロー

API バージョン互換確認後、以下のフローで処理する。

```
アプリ起動
    ↓
API バージョン互換性チェック（1.5.1）
    ↓ 互換あり
ローカルに認証トークンが存在するか
    ├── YES → 自動ログイン（サーバーでトークン検証・ユーザー情報同期）→ ホームへ遷移
    └── NO  → アカウント作成モーダルを表示（1.5.3）
```

- 自動ログイン中はローディングインジケーターを表示する
- サーバーエラー・通信失敗時: エラーメッセージ ＋ 再試行ボタンを表示する
- BAN されているアカウント: ログインを拒否し「アカウントが利用停止されています」を表示する

### 1.5.3 アカウント作成モーダル

ローカルに認証トークンが存在しない場合に表示するモーダル。

**モーダルの構成（上から順に）:**

1. アプリ名・ロゴ
2. 生年月日入力（年・月・日のピッカー）
   - ラベル: 「生年月日を入力してください（年齢確認のため）」
   - デフォルト: 未選択
   - **13 歳未満の場合:** ピッカー確定と同時に「このサービスは 13 歳以上の方を対象としています。」を表示し、以降の操作をブロックする（ソーシャルサインインボタンは表示しない）
3. 同意チェックボックス（デフォルト: オフ）
   - 「[利用規約]に同意する」（「利用規約」はリンクテキスト）
   - 「[プライバシーポリシー]に同意する」（「プライバシーポリシー」はリンクテキスト）
   - リンクをタップ → 外部ブラウザで各ページを開く
   - URL は `Assets/Scripts/Constants.cs` に以下の定数として定義:
     - `TermsOfServiceUrl = "https://example.com/terms-of-service"`（仮 URL）
     - `PrivacyPolicyUrl = "https://example.com/privacy-policy"`（仮 URL）
4. ソーシャルサインインボタン（縦並び）
   - **生年月日が 13 歳以上かつ両チェックボックスが ON の場合のみ有効**（それ以外はグレーアウト・タップ不可）
   - Google サインイン / Apple サインイン

**対応プロバイダー:**

| プロバイダー | 対応プラットフォーム | 備考 |
|---|---|---|
| Google サインイン | iOS / Android / PC | |
| Apple サインイン | iOS / PC（Mac） | App Store ガイドラインにより iOS では必須 |

**サインイン後の分岐:**

- 対応アカウントが既存 → ユーザー情報同期 → ホームへ遷移
- 新規アカウント → @name 設定画面へ遷移（1.5.4）

- × ボタンなし・モーダル外タップ無効（必ずサインインが必要）

**年齢確認の実装方針:**

- 生年月日をサーバーに送信し、サーバー側で年齢を計算して `age_group` フラグのみを保存する。生年月日は計算後に即時破棄し DB には保存しない（GDPR データ最小化原則への準拠）
- クライアント側でも 13 歳未満を即時ブロック（UX 補助・二重防護）
- App Store / Google Play のレーティングを 17+ に設定することと組み合わせて対応する（`docs/human-asset-todo.md` 参照）

**サーバー側の年齢判定:**

| 年齢 | `age_group` | 追加処理 |
|---|---|---|
| 13 歳未満 | — | `403 age_restricted` を返す。アカウント未作成 |
| 13〜15 歳 | `young_teen` | アカウント作成後、保護者同意フローを開始（セクション 1.5.6） |
| 16〜17 歳 | `teen` | 通常のアカウント作成 |
| 18 歳以上 | `adult` | 通常のアカウント作成 |

### 1.5.4 @name 設定画面（初回）

新規アカウント作成後に表示する。@name を設定しない限りアプリ本体（ホーム画面）に遷移できない。

- 3〜15 文字・英数字（a-z / A-Z / 0-9）・アンダースコア（`_`）のみ使用可
- 入力中にリアルタイムでバリデーション表示（文字数・使用可否）
- 重複チェックはサーバー側で実施（送信時）
- 設定完了ボタン → ホームへ遷移

### 1.5.5 複数プロバイダー連携

- 1 つのアカウントに複数のソーシャルプロバイダーを連携できる（例: Google と Apple を同一アカウントに紐づけ）
- 連携・解除の管理はユーザー設定画面（セクション 10）から行う
- 連携中のすべてのプロバイダーを解除することはできない（最低 1 つは維持必須）
- **拡張性:** 将来的なメールアドレス＋パスワード認証・SMS 認証の追加を考慮し、認証プロバイダーを抽象化したアーキテクチャで実装する（現時点では不要）

### 1.5.6 保護者同意フロー（young_teen のみ）

`age_group = 'young_teen'`（13〜15 歳）のアカウントは、保護者同意が完了するまで**アプリのすべての機能へのアクセスを完全にブロック**する。COPPA・GDPR 第 8 条への準拠を目的とする。TikTok・Snapchat・Discord 等の主要 COPPA 対応サービスと同じ完全ブロック方針を採用する（制限モードは設けない）。

**同意待ち中の画面（完全ブロック）:**

@name 設定完了後、保護者同意が完了するまでアプリはこの画面のみを表示する。ナビゲーションバー・その他 UI はすべて非表示。

画面の構成（上から順に）:
1. 「このサービスを利用するには保護者の同意が必要です」の説明文
2. 保護者のメールアドレス入力欄
3. 送信ボタン → 入力したメールアドレスへ確認メールを送信
4. 送信後: 「〇〇〇@xxx.com に確認メールを送りました。保護者の方に確認してもらってください。」を表示
5. 再送ボタン（1 日 1 回まで）・メールアドレス変更リンク
   - 当日の再送上限に達している場合: ボタンをグレーアウトし「本日の再送上限に達しました。明日また試してください。」を表示
   - リンク期限切れ（72 時間超過）を理由とする再送は 1 日 1 回制限を免除する（サーバーは `reason=link_expired` パラメータを受け付け、当日分の制限をスキップして送信する）

保護者のメールアドレスは確認メール送信後に保存しない。

**確認メールの仕様:**

- 件名: 「お子様の「[アプリ名]」利用についての同意確認」
- 本文: サービス概要・利用規約・プライバシーポリシーへのリンク・同意ボタン
- 同意ボタン: ワンクリックで `parental_consent_verified_at` を記録
- リンクの有効期限: 72 時間（期限切れ後は再送が必要）

**タイムアウト（同意未完了によるアカウント削除）:**

同意が一定期間完了しなかった場合、アカウントを削除する（COPPA・GDPR データ最小化原則・2025 年 COPPA 改正の保持期間明示要件への準拠）。

| タイミング | 処理 |
|---|---|
| アカウント作成から 7 日後（同意未完了の場合） | リマインドメールを保護者のメールアドレス宛に送信（「あと 7 日で登録が取り消されます」） |
| アカウント作成から 14 日後（同意未完了の場合） | アカウントを即時物理削除（`active_users` + アップロードファイル全件。ソフトデリートの 30 日猶予なし）。ユーザーは次回起動時に「登録期限が過ぎました。最初からやり直してください。」画面を表示 |

14 日という期限は Microsoft（GDPR 対応・2018 年実装）の業界事例に基づく。2025 年 COPPA 改正で義務化された書面による保持期間スケジュールの対象。

**同意完了後:**

- `parental_consents` テーブルに同意タイムスタンプを記録（監査証跡として 5 年保持。詳細: `api-abstract.md`）
- `active_users.parental_consent_verified_at` にタイムスタンプを記録
- タイムアウトジョブをキャンセル
- ホーム画面へ自動遷移

同意完了の検知: クライアントはアプリ起動時 / フォアグラウンド復帰時の `GET /startup` レスポンスで `parental_consent_status` フィールドを確認する。ポーリングは行わない。

**API エンドポイント:**

```
POST /api/v1/me/parental-consent/send
  body: { "email": "parent@example.com", "reason": "link_expired" }  ← reason は省略可
  → 確認メール送信。メールアドレスは送信後即時破棄（DB に保存しない）
  → 1 日 1 回制限: 超過時 429（ただし reason=link_expired の場合は制限を免除）
  → アドレス変更時は旧 JWT を無効化し新しい JWT を発行（旧リンクが無効になる）

GET  /api/v1/parental-consent/verify?token=<jwt>
  → 認証不要エンドポイント（ブラウザで直接アクセス）
  → JWT 検証後 parental_consent_verified_at を記録・タイムアウトジョブをキャンセル
  → 成功時: 「同意が完了しました。アプリをご確認ください。」HTML を返す
  → 失敗 / 期限切れ: 「リンクが無効または期限切れです。アプリで再送してください。」HTML を返す
```

**実装:**

保護者同意トークンは JWT（有効期限 72 時間・`sub` = user_id・`jti` = UUID でアドレス変更時の無効化に使用）で発行し、DB には保存しない。`parental_consents` テーブルへの記録は同意完了時のみ行う。

### 1.5.7 セキュリティ通知モーダル（大規模データ侵害時）

`/startup` API レスポンスの `securityNotice` フィールドが `null` でない場合、ホーム画面への遷移前に強制表示するモーダル。GDPR Art. 34 の「直接通知（directly）」要件を満たすための手段。

**表示タイミング:**
- ログイン完了・`/startup` レスポンス処理後、ホーム画面への遷移直前
- 未確認の通知がある場合のみ表示（確認済みユーザーには表示しない）
- 自動ログイン・手動ログインいずれの場合も同様

**モーダルの構成:**

| 要素 | 仕様 |
|---|---|
| タイトル | `securityNotice.title`（例: 「重要なセキュリティに関するお知らせ」） |
| 本文 | `securityNotice.body`（侵害内容・影響範囲・対処済み措置・ユーザーへの推奨行動） |
| ボタン | 「確認しました」1 つのみ（縦並び不要） |
| × ボタン | **なし**（必ず確認操作が必要） |
| モーダル外タップ | **無効** |

**確認後の処理:**
1. `POST /api/v1/me/security-notice-ack` を送信（`noticeId` を含む）
2. レスポンス成功後にホーム画面へ遷移
3. API 失敗時はリトライダイアログを表示（ネットワーク断等）

## 2. ワールドモード

### 2.1 概要

- ワールドを3D空間で自由に移動・探索する
- 同じルーム（ワールドインスタンス）に参加しているユーザー同士でVivox 3D位置音声通話が可能
- ワールドに入る前にアバターを選択する

### 2.2 アバター選択

| 選択元 | 内容 |
|---|---|
| 自分のスロットのアバター | テクスチャ編集・アクセサリ込みの改変状態で表示 |
| 購入したアバター（直接利用） | 素の状態のまま（テクスチャ編集・アクセサリ取り付け不可） |

### 2.3 発話インジケーター

- ユーザーが発話中: アバターの頭上に発話インジケーターアイコンが点灯する

### 2.4 ユーザー情報パネル

アバターをタップしたとき、またはルーム内メンバー一覧・ワールド詳細画面等でユーザーをタップしたときに表示する共通パネル。表示コンテキスト（ルーム内外）によらず同一UIを使用する。

**ユーザー情報セクション:**

| 項目 | 内容 |
|---|---|
| アバターアイコン（大） | 現在使用中アバターのアイコン |
| 表示名 | ユーザーの表示名。公認バッジ保持者の場合は表示名の右に公認チェックマークアイコン（`icon_verified.png`）を表示 |
| @name | @name 形式（検索用） |
| 称号アイコン | 有効かつ表示 ON のもの一覧 |
| フォロワー数 / フォロー中数 | 数値表示。タップでフォロー管理モーダルを表示（後述） |
| ショップリンクボタン | ショップクリエイターの場合のみ表示 |

**アバター情報セクション:**

- 使用中アバターが購入アバターの場合: アバター名 + ショップへのリンクボタン
- テクスチャ改変済みスロットアバターの場合: ベースアバターのショップへのリンクボタン

**アクションセクション（自分自身の場合は全て非表示）:**

| 要素 | 内容 |
|---|---|
| フォローボタン / フォロー解除ボタン | フォロー状態に応じて切り替え |
| フレンド申請ボタン / フレンド解除ボタン / 「申請済み」 | フレンド状態に応じて切り替え。非表示中ユーザーには非表示 |
| 「…」ボタン | タップで「非表示にする / 非表示解除」「ブロックする / ブロック解除」「通報する」を縦に表示 |

**非表示にする**: 確認モーダルを表示する（セクション 1.1 共通モーダルUIルール準拠）

> **「このユーザーを非表示にしますか？」**
> ［非表示にする］　［キャンセル］

**ブロックする**: 確認モーダルを表示する

> **「@name をブロックしますか？」**
> ブロックすると、相手のアバターが見えなくなり、音声も聞こえなくなります。フレンド・フォロー関係も解除されます。相手にはブロックしたことは通知されません。
> ［ブロックする］　［キャンセル］

- ブロック実行後、パネルを閉じる
- ブロック済みユーザーのパネルには「ブロック解除」ボタンを表示する（「…」メニュー内）
- ブロック一覧の確認・解除は設定タブのアカウント設定（セクション 5）から行う

#### フォロー管理モーダル

フォロワー数 / フォロー中数をタップすると、ルームを離れずにモーダルシート（下からスライド）でフォロー管理を表示する。

- モーダル内でそのユーザーのフォロワー一覧 / フォロー中一覧を切り替え表示する
- 一覧内のユーザーをタップすると**モーダルをスタック**してそのユーザーの情報パネルを表示する（戻るボタンで前のモーダルに戻る）
- モーダル外タップまたは最下層の戻るボタンでモーダルを閉じる

### 2.5 バックグラウンド通話モード（プレミアム限定）

| 項目 | 内容 |
|---|---|
| 対象プラットフォーム | iOS / Android のみ |
| 機能 | 3D描画をOFFにし、バックグラウンドおよびスリープ中も通話を継続 |
| 非プレミアム時の挙動 | バックグラウンド移行またはスリープ時に通話切断 |

実装: `OnApplicationPause` イベントを利用し、プレミアムユーザーの場合のみバックグラウンド処理を継続。Vivox SDKのバックグラウンド動作設定と組み合わせて対応する。

### 2.6 タッチ操作

| 操作エリア | 操作 | 動作 |
|---|---|---|
| 画面下半分 | スライド | キャラクター移動（詳細は 2.6.1） |
| 画面下半分 | ダブルタップ | ジャンプ（詳細は 2.6.2） |
| 画面上半分 | スライド | カメラを回転する（水平: 左右回転 / 垂直: 仰角調整） |

PC（Windows / Mac）ではキーボード（WASD / 矢印キー）での移動・スペースキーでのジャンプ・Shift キーでのスプリント・マウスドラッグでのカメラ回転に対応する。

#### 2.6.1 移動・スプリント入力

- **移動開始**: 画面下半分でスライド開始と同時に、スライド方向へキャラクターが移動する
- **通常移動 vs スプリント**: スライド開始から **0.3 秒以内**にスライドの移動量が誤差範囲内に収まった場合（指が止まった状態でタッチを継続）、スプリントに移行する。0.3 秒を超えてもスライドが続いている場合は通常移動のまま
- **移動方向**: スプリント移行後の移動方向はスプリント移行直前のスライド方向を維持する
- **入力の優先順位（マルチタッチ）**: 画面下半分への移動スライドが複数同時に発生した場合、**最初のスライド入力のみ**を移動入力として採用する。最初のタッチが離れるまで後続のスライド入力は移動に影響しない

#### 2.6.2 ジャンプ

- **入力**: 画面下半分を**ダブルタップ**するとジャンプする
- **ジャンプ高さ**: **0.55 単位**（高さ 0.5 のブロックには乗れるが、高さ 0.75 のブロックには乗れない）
- **方向ジャンプ（移動中）**: 移動スライド入力をしている状態で別の指でダブルタップすると、移動方向へのジャンプになる
- **方向ジャンプ（ダブルタップ→スライド）**: ダブルタップの 2 回目のタップをそのままスライドに移行した場合、そのスライド方向へのジャンプになる
- **設定によるボタン表示時**: ダブルタップによるジャンプは無効化される（後述セクション 19.6）

#### 2.6.3 コントロールボタン（オプション表示）

設定（セクション 19.6）でジャンプボタン・スプリントボタンを画面に表示できる。

| ボタン | 配置 | 動作 |
|---|---|---|
| ジャンプボタン | 画面下部（Safe Area 内） | タップでジャンプ |
| スプリントボタン | 画面下部（Safe Area 内） | タップするたびに通常移動 ↔ スプリント移動を切り替え |

- ボタン表示時は**ダブルタップによるジャンプ**と**スライド速度によるスプリント移行**が無効化される
- デフォルトは両ボタンとも非表示

### 2.7 撮影機能

インワールド HUD 右上のカメラボタンをタップすると撮影モードに移行する。撮影モード中はキャラクター移動操作を無効化し、通常の HUD（セクション 2.11.1）を非表示にして撮影専用UIに切り替える。

#### 2.7.1 撮影モード HUDレイアウト

| 位置 | 要素 |
|---|---|
| 右上 | × ボタン（撮影モード終了・ワールドモードに戻る） |
| 下中央 | 撮影ボタン（シャッター） |
| 右下 | スタンプアイコンボタン |
| 左下 | 直近の撮影サムネイル（撮影後に表示・タップで拡大） |

#### 2.7.2 カメラ操作

| 操作 | 通常モード | 撮影モード |
|---|---|---|
| 画面上部 1 本指スライド | カメラ回転（水平/垂直） | カメラ回転（水平/垂直）（同じ） |
| 画面上部 2 本指操作 | — | **撮影モード専用**: ピンチで拡大縮小・スライドで平行移動（カメラ位置を変更） |

- 撮影モード終了時（× ボタン押下）: カメラ位置を撮影モード開始直前の相対位置に復元する

#### 2.7.3 スタンプ機能

スタンプはカメラ映像に重畳して配置する。撮影前に自由に配置でき、撮影画像に焼き込まれる。

**スタンプ配置フロー:**

1. スタンプアイコンボタンをタップ → スタンプ選択メニューが画面下から出る
2. スタンプをメニュー外（画面上部）へドラッグ → メニューが閉じ、スタンプが画面上に配置される

**配置済みスタンプの操作:**

| 操作 | 動作 |
|---|---|
| 1 本指ドラッグ | スタンプを移動 |
| 2 本指回転 | スタンプを回転 |
| ピンチイン・アウト | スタンプを拡大縮小 |
| ドラッグ中に画面下中央のゴミ箱アイコンへ重ねる | スタンプを削除 |

- ドラッグ中のみゴミ箱アイコンを画面下中央に小さく表示する
- スタンプは複数配置可能

**スタンプ状態の保持:**

- ルームセッション中はスタンプの配置・サイズ・回転をメモリに保持する
- 撮影モードを終了して再度入るたびに、前回の配置状態を復元する
- ルーム退室時またはアプリ終了時にリセットされる

**スタンプ選択メニュー:**

| タブ | 内容 | 利用条件 |
|---|---|---|
| デフォルト | 初期同梱スタンプ | 全ユーザー利用可 |
| プレミアム | プレミアム限定スタンプ | プレミアム会員のみ。非プレミアムはロックオーバーレイ表示・利用不可 |
| 購入グループ（複数）| ショップで購入したスタンプグループ（グループ単位で購入）| 購入ユーザーのみ |

- タブの並び順: デフォルト → プレミアム → 購入グループ（購入が新しい順）
- デフォルトで開くタブは「デフォルト」グループ
- セッション中に別タブを選択した場合: 次回メニューを開いたときも最後に選択したタブを表示する

**プレミアム限定の特殊スタンプ:**

プレミアムグループには以下の特殊仕様スタンプが含まれる。

| 種別 | 説明 |
|---|---|
| 色変えスタンプ | 配置後に色を変更できるスタンプ。スタンプメニューのサムネイル右下にカラーサークルを表示して識別する |
| 文字入れスタンプ | テキスト入力が可能な色変えスタンプ（色変えスタンプの一種）。サムネイル右下にもカラーサークルを表示する |

文字入れスタンプは以下の 3 種類:

| # | 背景 | 文字色 |
|---|---|---|
| 1 | 透明 | 変更可能（色変えで変更） |
| 2 | 角丸四角（背景色変更可能） | 白固定 |
| 3 | 角丸四角（背景色変更可能） | 黒固定 |

**文字入れスタンプのテキスト編集:**

1. ドラッグ＆ドロップで配置 → テキスト入力状態になる（キーボードが表示される）
2. 文字の外をタップ → テキスト入力完了（キーボードが閉じる）
3. スタンプをタップ → 再度テキスト入力可能

**色変えスタンプのカラーパレット:**

- 配置直後、画面下部にカラーパレットが表示される
  - 文字入れスタンプの場合: キーボード直上に表示（文字編集開始と同タイミング）
- 色鉛筆と同じ 12 色を横一列（左 → 右）で並べる
- 画面端を超える場合は横スクロール可能（スクロールバー非表示）
- **一番左の要素はスポイトボタン**（スポイトアイコン）
- カラーパレットの色をタップ → スタンプの色が変わる
- スタンプでもカラーパレットでもない部分をタップ → 色選択完了・パレット非表示
- スタンプを再タップ → カラーパレットを再表示（文字入れスタンプは文字編集と同タイミング）

**スポイトモード:**

カラーパレットのスポイトボタンをタップするとスポイトモードに移行する。

- 移行時: スタンプ・カラーパレットを含む他のUIをすべて非表示にする
- 画面中央: 円形のスポイトカーソルを表示する
- 画面中央上部: 現在サンプリング中の色を円形で表示する（外側を白で縁取る）
- **スポイト操作フロー:**
  1. 画面をタップ → タップ位置の色をサンプリングする（色表示サークル自体は貫通して背景色を取得）→ 色表示サークルが更新される
  2. 指を離して 0.4 秒後 → 色確定・スポイトモード終了・カラーパレット表示に戻る（選択色が反映）
  3. 0.4 秒以内に再タップ → 再度スポイト開始（ステップ 1 に戻る）

#### 2.7.4 撮影・プレビュー

- 撮影ボタンをタップするとシャッター音が鳴り、スタンプ込み・UI なしの画像を撮影する
- **撮影と同時に端末のカメラロールへ保存する**（iOS: `Photos API` / Android: `MediaStore API`）
- 撮影直後、左下に小さいサムネイルプレビューを表示する
  - サムネイルをタップ → 拡大表示
  - 拡大表示中にもう一度タップまたは右上 × ボタン → 拡大表示を閉じる

### 2.8 コライダー仕様（キャラクター）

| 種別 | 形状 | 直径 | 高さ |
|---|---|---|---|
| 物理コライダー | カプセル | 0.4m | 1.45m |
| タッチ判定コライダー | カプセル | 1.0m | 2.0m |

- 物理コライダー: 壁・床などワールドジオメトリとの接触判定のみに使用。**プレイヤー同士の衝突判定は行わない**（アバター同士は重なることができる）
- タッチ判定コライダー: タップによるアバター選択（アバタータップUI表示のトリガー）に使用

実装: Unity の Physics Layer Collision Matrix で `Player` レイヤー同士の衝突を無効化する。`Player` レイヤーは `World` レイヤーとのみ衝突する。

#### 2.9.1 ホームタブ

ワールドの発見・検索を行うメインタブ。以下のロジックで表示コンテンツを構成する。

---

##### 通報フィルター（全経路共通）

72時間以内に **異なるユーザーから3件以上の通報** を受けたワールドはホームのいかなる経路にも表示しない。

- `worlds.recent_report_count` カラム（キャッシュ値）で絞り込む（`< 3` のみ返す）
- このカラムは通報発生時にインクリメントし、72時間バッチで期限切れ通報をデクリメント/クリアする

---

##### フォロー中・フレンドの新着（同期・上位固定）

24時間以内に **フォロー中またはフレンドのユーザーが公開したワールド** で、かつ **自分がいいねも入室もしていない** ものを同期的に取得してリスト最上位に表示する。

- APIリクエスト時に都度取得（非同期キャッシュなし）
- 複数件ある場合は公開日時の新しい順

---

##### メインフィード（アクティブ度による分岐）

フォロー新着の下に表示するメインフィードは、直近72時間以内のログイン履歴の有無で動作が変わる。

**パターンA: 非アクティブユーザー（72時間以内ログインなし）**

専用レコメンドは生成しない。**24時間以内のいいね数上位**（自分がいいねも入室もしていないワールド）をランダムな順番で返す。

**パターンB: アクティブユーザー（72時間以内にログインあり）**

専用レコメンドと24時間いいね上位を **2:1** の割合で混在させてランダムな順番で返す。

- 専用レコメンドが全件既読になった場合は24時間いいね上位のみを返す（パターンAと同じ動作）

---

##### 専用レコメンド（非同期・1日1回生成）

アクティブユーザーに対し、バックグラウンドジョブで1日1回生成する。生成結果は `user_recommendations` テーブルに保存し、クライアントがフィードを取得した際に既読フラグを付ける。既読フラグが付いたレコメンドはフィードに返さない。

**生成手順:**

1. **シード収集**: 直近72時間の自分のいいね履歴を新しい順に最大10件取得する。10件未満の場合は入室記録も合わせて合計10件になるよう補完する（いいね優先）。

2. **おすすめタグ・おすすめユーザー導出**:
   - シードワールドに付いている全タグ → **おすすめタグ**（共起ベース。将来的にセマンティック類似度に拡張予定）
   - シードワールドの作成者 → **おすすめユーザー**

3. **おすすめタグマッチワールドの収集**（いいねも入室もしていないワールドのみ）:

   | 条件 | 最大件数 |
   |---|---|
   | おすすめタグを持つワールド・24時間以内公開の新着順 | 10件 |
   | おすすめタグを持つワールド・24時間以内の入室数順（24時間入室数 ≥ 6 のみ） | 20件 |
   | おすすめタグを持つワールド・24時間以内のいいね数順（24時間いいね数 ≥ 6 のみ） | 20件 |
   | おすすめタグを持つワールド・累計いいね数順（累計いいね数 ≥ 12 のみ） | 20件 |

4. **おすすめユーザーワールドの収集**（いいねも入室もしていないワールドのみ）:
   - おすすめユーザーのワールドを累計いいね数順（累計いいね数 ≥ 6 のみ）で最大30件

5. **重複除去・シャッフル**: 3〜4で集めたリストから重複を除き、ランダムに並べたものを専用レコメンドとして保存する。

---

##### ホームタブ APIエンドポイント

| エンドポイント | 説明 |
|---|---|
| `GET /home/feed` | フォロー新着 + メインフィード（パターンA/B自動判定）。カーソルページネーション |
| `POST /home/feed/read` | レコメンドアイテムの既読マーク（フィード取得時に自動呼び出し） |

---

ワールド検索はホームタブから行う:

- 検索バー: ワールド名・ワールドID で検索
- タグフィルター: 横スクロールのタグチップ（複数選択可・AND 条件）
- ルームID で直接検索した場合、そのルームに直接参加できる

検索の実装:
- ワールド名検索は `pg_trgm` + GIN インデックスによる部分一致（`ILIKE '%query%'`）
- タグ検索は `tag_normalized` の完全一致。複数タグ選択時は AND 条件
- ワールド名検索とタグフィルターは同時適用可能

##### ルームタブ

| セッション残り時間 | 例: `残り 1:23:45`（時:分:秒）。通常ユーザー上限 90 分 / プレミアム 12 時間 |

**旧バージョン警告**: 現在のルームが使用するワールドバージョンが最新でない場合、ルーム情報の上部に「このルームは旧バージョンのワールドを使用しています。新しいプレイヤーはこのルームに参加できません。」と警告を表示する。

**ルーム再入室ボタン**: タップすると一度ルームを退室し、スポーンポータル経由で同じルームに再入室する。ただし旧バージョンのルームへの再入室は不可（旧バージョン警告表示中はボタンを無効化し「旧バージョンのルームには再入室できません」を表示）。

**ルームメンバー一覧**:

- 現在ルームに参加しているメンバーをアバターアイコン + 表示名で一覧表示
- 非表示にしたメンバーは一覧の下部にアイコン・表示名を薄く表示する
- メンバーをタップするとセクション 2.4 のユーザー情報パネルを表示する

- 「非表示にする」を選択: クライアント側でアバターレンダリング・Vivox ボイス再生を停止。メンバー一覧の下部に薄く表示したままにする。「非表示にしました」フラッシュメッセージを表示

**通報する**: 通報モーダルを表示する（セクション 6.5 の共通仕様に準拠）

### 2.12 ルームセッション制限

#### セッション時間上限

| ユーザー種別 | 連続在室上限 |
|---|---|
| 通常ユーザー | 90分 |
| プレミアム会員 | 12時間 |

- セッション時間はルーム入室時点からカウント開始
- 残り **10分・5分・1分** のタイミングでフラッシュメッセージ（画面上部トースト）で通知する
- 制限時間到達時: 「制限時間に達しました」ダイアログを表示し、OK タップまたは数秒後に自動でルームを退室する
- 退室後は同じルームに即座に再参加可能（再参加後はセッション時間がリセットされる）

#### 放置自動退室（通常ユーザーのみ）

- 操作なしで **10分**経過するとルームを自動退室する
- 以下のいずれかがあれば「操作あり」と判定しタイマーをリセットする:
  - タッチ・タップ操作
  - キャラクター移動入力
  - UI 操作（メニュー等）
- 発話（マイク入力）は操作としてカウントしない
- 放置退室は告知なしで即時退室する
- プレミアム会員はバックグラウンド通話が可能なため、放置自動退室の対象外

### 2.13 ネットワーク切断時の挙動（ルーム内）

ルームにいる状態で Netcode for GameObjects の接続喪失を検知した場合、以下のフローで処理する。

#### 自動再接続フロー

1. 切断を検知 → **再接続中モーダル**を表示する（他の UI は操作不可）
2. 指数バックオフで最大 5 回、同一ルームへの再接続を試みる:

| 試行 | 待機時間 |
|---|---|
| 1 回目 | 1 秒後 |
| 2 回目 | 2 秒後 |
| 3 回目 | 4 秒後 |
| 4 回目 | 8 秒後 |
| 5 回目 | 16 秒後 |

3. **再接続成功** → 通常の入室フローと同様にスポーンポータルからのスポーンをやり直す
4. **全試行失敗** → **接続失敗モーダル**に切り替える

#### 再接続中モーダル

- メッセージ: 「接続が切断されました。再接続しています…」
- ボタン: 「タイトルに戻る」（タップでタイトル画面へ遷移）
- × ボタンなし・モーダル外タップ無効

#### 接続失敗モーダル

- メッセージ: 「接続に失敗しました」
- ボタン: 「タイトルに戻る」（タップでタイトル画面へ遷移）
- × ボタンなし・モーダル外タップ無効

#### 再接続成功後の挙動

- スポーンポータルからのスポーンをやり直す（通常の入室フローと同一）
- ギミック状態はオーナーからのスナップショット受信で同期する（詳細: `docs/world-creation.md` セクション 9.9）

### 2.13 推奨公開ルーム参加ロジック

「公開ルームに参加」ボタンをタップすると以下の優先順位で自動参加先を決定する。ロジックはサーバー側で実行し、レース状態を防止する。

```
ステップ1: 自分の言語設定と一致する公開ルームのうち
          1人以上いて空きがあるルームが存在するか
  → YES: 空きのある中で最も人数が多いルームに参加
  → NO:  ステップ2へ

ステップ2: 自分の言語のルームが1つ以上存在し
          すべて満席か
  → YES: 自分の言語設定で新しい公開ルームを作成して参加
  → NO:  ステップ3へ

ステップ3: 英語（en）の公開ルームのうち
          1人以上いて空きがあるルームが存在するか
  → YES（自分の言語 = 英語）: 最も人数が多いルームに参加
  → YES（自分の言語 ≠ 英語）: モーダル「現在参加できるのは英語のルームのみです。参加しますか？」
       → OK: 最も人数が多いルームに参加
       → キャンセル: ステップ4へ
  → NO:  ステップ4へ

ステップ4: 自分の言語設定で新しい公開ルームを作成して参加
```

- 新規作成ルームの言語は作成者の言語設定を自動適用
- 参加先が決まるまでローディング表示（最大数秒）

### 2.14 その他ルーム画面

ワールド詳細の「その他のルームを見る」から遷移する。公開ルームの選択・各種ルームの新規作成ができる。

**公開ルーム一覧:**

| 項目 | 内容 |
|---|---|
| ルームID | 短縮ID |
| 言語 | ルームの設定言語（言語コードまたは国旗アイコン） |
| 参加人数 | `現在人数 / 最大人数` 形式（例: `3 / 6`） |
| 参加ボタン | タップで参加。満員ルームは無効化し「満員」バッジを表示。旧バージョンのルームは無効化し「旧バージョン」バッジを表示（タップ時に「このルームは旧バージョンのワールドを使用しているため参加できません」メッセージを表示） |

**表示順（言語優先ソート）:**

ビューワーの言語設定と一致するルームを上位に表示し、一致しないルームを下位に表示する。言語グループ内は人数の多い順。

```
表示順 =（言語一致: 0 / 不一致: 1）→ 人数の多い順
```

**ルーム作成ボタン:**

- 「新しい公開ルームを作成」→ 公開ルームを新規作成（言語は作成者の言語設定を自動適用。作成後にルーム内から変更可）
- 「フォロワー限定のルームを作成」→ フォロワー限定ルームを新規作成して参加する（言語は作成者の言語設定を自動適用）
- 「招待制ルームを作成」（プレミアム限定）→ 招待制ルームを新規作成

### 2.15 フレンドがいるルーム一覧画面

フレンドが現在参加しているルーム（全ワールド横断）を一覧表示する。フレンドリスト画面またはワールド詳細画面などからアクセスできる。

**表示項目:**

| フレンド | そのルームにいる自分のフレンドのアバターアイコン + ユーザー名 |
| ルームID | 短縮ID |
| 参加人数 | `現在人数 / 最大人数` 形式 |
| 参加ボタン | タップで参加（フレンドのみルームはフレンド関係を参加時にサーバー検証） |

## 3. アバター管理

ナビゲーションバーの「アバター管理」タブから遷移する。アバターとアクセサリの管理・編集を行う画面。

### 3.1 タブ構成

画面上部にアイコンタブを配置し、以下の 4 カテゴリを切り替える。

| タブ | 内容 |
|---|---|
| 編集アバター | スロットを利用した保存・編集済みアバター一覧 |
| 所持アバター | デフォルト（プリセット）・プレミアム限定・ショップ購入アバター一覧 |
| 編集アクセサリ | スロットを利用した保存・編集済みアクセサリ一覧 |
| 所持アクセサリ | デフォルト（プリセット）・プレミアム限定・ショップ購入アクセサリ一覧 |

**デフォルトタブ選択ルール:**

1. 編集アバタースロットにアバターが 1 つ以上ある → 「編集アバター」タブ
2. 編集アバターがなく、ショップ購入アバターが 1 つ以上ある → 「所持アバター」タブ
3. いずれもない（新規ユーザー等）→ 「所持アバター」タブ（プリセットが表示される）

### 3.2 編集アバタータブ

スロットを使って保存された改変アバターの一覧をサムネイルで表示する。

| 要素 | 内容 |
|---|---|
| スロット一覧 | サムネイル + アバター名を表示 |
| 空きスロット | 「追加」ボタンを表示（上限到達時は「上限到達」と表示し削除で解放を促す） |
| プレミアム解約後のロックスロット | ロックアイコン表示（データ保持・利用不可） |
| スロット上限 | 通常 10個 / プレミアム 最大100個 |

**追加ボタンをタップしたとき:**

「所持アバターから追加」「.vrm ファイルをアップロードして追加」の選択モーダルを表示する。

- 所持アバターから追加: 所持アバター一覧から選択 → 編集画面（スロットを 1 つ消費）
- .vrm ファイルをアップロード: ファイルピッカー → VRM バリデーション → 編集画面（スロットを 1 つ消費）

VRM アップロード時のバリデーション:
```
ファイルピッカー → VRMファイル選択
  ├── 仕様違反（容量超過 / ポリゴン超過 / ボーン超過 / VRM 1.0以外）
  │     └── エラーメッセージ表示（処理中断）
  └── 仕様OK → テクスチャを第1レイヤーに反映 → 編集画面
```

**スロットをタップしたとき:**

| 操作 | 内容 |
|---|---|
| 利用アバターに設定 | ワールドモード入室時のデフォルト利用アバターに設定する |
| 編集 | アバター編集画面（セクション 3.5）に遷移 |
| 削除 | 確認モーダル表示後にスロットを解放（データ削除） |

### 3.3 所持アバタータブ

デフォルト（プリセット）・プレミアム限定・ショップ購入アバターの一覧をサムネイルで表示する。

**アバターをタップしたとき:**

| 操作 | 内容 |
|---|---|
| 直接利用 | このアバターをワールドモードで直接利用する（スロット消費なし・テクスチャ改変不可） |
| 編集スロットに追加して編集 | 編集スロットが空いていればスロットを 1 つ消費して編集画面へ遷移。スロット満杯の場合はエラー表示 |

### 3.4 編集アクセサリタブ・所持アクセサリタブ

アバタータブと同様の構成。アクセサリ固有の仕様:

- スロット上限: 通常 10個 / プレミアム 最大100個
- 追加モーダル: 「所持アクセサリから追加」「.glb ファイルをアップロードして追加」

### 3.5 アバター編集画面

編集アバタースロットから「編集」操作で遷移する。画面上部に 3D プレビューを常時表示し、画面下部にアイコンタブを配置する。

#### 3.5.1 テクスチャタブ

アバターのテクスチャ編集を行う（仕様詳細: `docs/unity-game-abstract.md` セクション8）。

**保存時のAPIアップロード:**

1. レイヤー画像（変更済みのもののみ、256×256）
2. レイヤー構造JSON
3. 統合画像（256×256）

#### 3.5.2 アクセサリタブ

アバターに取り付けるアクセサリを管理する。最大4個まで取り付け可能。

**未選択時 — アクセサリ一覧（4 サブタブ）:**

| サブタブ | 内容 |
|---|---|
| 利用中 | 現在アバターに取り付けているアクセサリ |
| 保存・編集 | 編集アクセサリスロットに保存済みのアクセサリ |
| ショップ購入 | ショップで購入済みのアクセサリ |
| デフォルトプリセット | 初期提供のプリセットアクセサリ |

- 利用中アクセサリをタッチ → そのアクセサリが選択状態になる
- 利用中以外のアクセサリをタップ、または 3D プレビューにドラッグアンドドロップ → アクセサリ編集画面（セクション 3.6）で設定されたデフォルト位置でアバターに取り付けて選択状態になる
- 選択中のアクセサリはサムネイルにチェックマークが表示される

**選択状態時 — アクセサリタブに追加表示される要素:**

| 要素 | 内容 |
|---|---|
| 設置ボーン | 取り付け位置の選択フォーム（頭 / 右手 / 左手 / 右足 / 左足 / 胸 / 腰） |
| 削除ボタン | アバターからアクセサリを取り外す（スロットデータは保持） |

- 選択状態でギズモが表示される。上部中央のボタンで移動 / 拡大縮小 / 回転を切り替えてアタッチ位置を調整できる（ボーンからの相対位置）
- 選択解除: アクセサリでもアバターでも UI でもない部分をタップ、または選択中アクセサリのサムネイルをもう一度タップ

### 3.6 アクセサリ編集画面

編集アクセサリスロットから「編集」操作で遷移する。画面上部に 3D プレビューを常時表示し、画面下部にアイコンタブを配置する。

#### 3.6.1 テクスチャタブ

アクセサリのテクスチャ編集を行う（仕様詳細: `docs/unity-game-abstract.md` セクション8）。

**保存時のAPIアップロード:**

1. レイヤー画像（変更済みのもののみ、64×64）
2. レイヤー構造JSON
3. 統合画像（64×64）

#### 3.6.2 配置タブ

アバターへのデフォルト取り付け位置とボーンを設定する。

- **配置タブ選択時**: 3D プレビューが拡大され、自分のデフォルトアバターとともにアクセサリが表示される
- タブ部分は最小サイズで表示し、以下の要素のみを配置する:

| 要素 | 内容 |
|---|---|
| つける場所 | アタッチ先の選択フォーム（頭 / 右手 / 左手 / 右足 / 左足 / 胸 / 腰） |
| アバター切り替えボタン | ワールドでのアバター選択と同様の操作でプレビューに使用するアバターを変更できる |

- ギズモ（上部中央ボタンで移動 / 拡大縮小 / 回転を切り替え）で取り付け位置をボーンからの相対位置として調整できる
- ここで設定した位置がアバター編集画面（セクション 3.5）でアクセサリを取り付けたときのデフォルト位置になる

### 3.7 編集可否フラグ

| フラグ | テクスチャ編集 | アクセサリ取り付け |
|---|---|---|
| Edit OK | 可（レイヤーも利用可） | 可 |
| Edit NG | 不可 | 可 |

- プリセット・ローカル VRM ファイルは常に **Edit OK** 扱い
- ショップで購入したアバターは商品登録時の設定に従う

## 4. ショップモード

### 4.1 概要

- ゲーム内コインを使ってアバター・アクセサリ・ワールドオブジェクト・スタンプを購入する

## 5. ユーザー情報・称号システム

### 5.1 ユーザーID（@name）

- `@name` 形式
- **使用可能文字**: 英数字（a-z / A-Z / 0-9）・アンダースコア（`_`）のみ（大文字小文字は保存時に小文字へ正規化）
- **文字数**: 3〜15 文字
- 全ユーザーで一意
- ユーザー検索・アバタータップ UI・違反報告などに利用

**変更ルール:**

| 条件 | 内容 |
|---|---|
| 初回設定 | アカウント作成時に全ユーザーが 1 回設定（セクション 1.5.2） |
| 変更可否 | **プレミアム会員のみ**変更可能（通常ユーザーは変更不可） |
| 変更頻度 | 前回の変更（または初回設定）から **90 日（約 3 カ月）以上**経過している場合のみ変更可 |
| 古い @name の扱い | 変更後は即時解放される（他ユーザーが取得可能・リダイレクト・予約なし） |

**変更 UI（ユーザー設定画面内）で以下を明示する:**

- 現在の @name
- 次回変更可能日（プレミアム会員のみ表示）
- 「古い @name は変更後すぐに解放され、他のユーザーが取得可能になります。リダイレクトはサポートされません。」という注意文言
- 通常ユーザーには「@name の変更にはプレミアム会員が必要です」と表示し、変更フォームをロック

### 5.2 称号一覧

| 称号 | 表示条件 | 付与方法 |
|---|---|---|
| プレミアム | プレミアム会員中 | 自動付与 |
| ショップ開設者 | ショップ登録済み | 管理画面でショップとユーザーを紐づけた時点で自動付与 |
| 開発者 | 開発チームメンバー | 管理画面から手動付与 |
| 運営関係者 | 運営スタッフ | 管理画面から手動付与 |

- 現状、ショップの新規開設は管理画面経由のみ（ユーザー自己申請不可）
- ショップクリエイター情報はユーザーアカウントに紐づける
- 各称号の表示/非表示はユーザー設定から個別に切り替え可能

### 5.3 公認バッジ

インフルエンサー・著名人など、本物であることを証明したい特定ユーザーに付与する公認バッジ。称号とは独立した概念で、ユーザーによる表示/非表示の切り替えはできない（常に表示）。

| 項目 | 仕様 |
|---|---|
| 表示位置 | **表示名の直右**（称号アイコン一覧より前）にチェックマークアイコン（`icon_verified.png`）を表示 |
| 表示場所 | ユーザー情報パネル・アカウント情報画面・ルームメンバー一覧・フレンド一覧・フォロー/フォロワー一覧・ワールド内アバター頭上名前タグなど、表示名が出るすべての場所 |
| 付与/剥奪 | 管理画面（`admin` / `super_admin` ロール）からのみ操作可能。`moderator` は不可 |
| DB | `active_users.is_verified BOOLEAN NOT NULL DEFAULT false` |
| ユーザー操作 | なし（自己申請・表示切り替え不可） |

## 6. 違反報告・モデレーションシステム

### 6.1 通報の導線

| 対象 | 導線 |
|---|---|
| ユーザー | ワールドモードでアバタータップ → 「違反報告」ボタン / アカウント情報パネルの「通報する」ボタン |
| ワールド | ワールド詳細画面のその他アクションボタン（…）→「通報する」 |
| ショップ商品 | 商品詳細画面のその他アクションボタン（…）→「通報する」 |

いずれの導線からも共通の通報モーダル（セクション 6.5）を開く。通報内容は管理画面の違反報告一覧（セクション 8.4）に集約される。

### 6.2 アバターBAN基準

以下に該当するアバターはBAN対象とする:

| カテゴリ | 内容 |
|---|---|
| 露出 | 水着等、過度な露出を含む表現 |
| 性的表現 | 性的な表現を含む |
| ゴア・残虐 | ゴア・残虐表現を含む |
| 動作問題 | アプリの動作に支障を与える（過負荷・クラッシュ等） |
| 規約違反 | ファイルサイズ・ポリゴン・ボーン数等の技術規約違反 |
| 違法助長 | 違法行為を助長する可能性のある表現 |

### 6.3 制裁ポリシー

```
管理画面での審査
  ├── 問題なし → 承認
  └── 違反あり
        ├── 通常違反 → アバターBAN ＋ ユーザーへ警告発行（Warning +1）
        └── 重大違反（違法性高等）→ 「警告のみ」または「即時ユーザーBAN」を選択

累積警告回数 ≥ 2 → 原則ユーザーBAN
```

### 6.4 BAN後の処理

| 対象 | 処理内容 |
|---|---|
| アバターBAN | 該当アバターをサービス上で非表示・利用不可 |
| ユーザーBAN | ログイン不可・全データ凍結（物理削除なし） |

### 6.5 通報モーダル共通仕様

ユーザー・ワールド・ショップ商品いずれの通報でも同一モーダルを使用する。モーダル共通UIルール（セクション 1.1）に準拠する。

**モーダルの構成:**

1. **通報理由の選択**（必須）: 以下から1つを選択する

   | 選択肢 | 詳細入力 |
   |---|---|
   | スパム・迷惑行為 | 任意 |
   | ヘイト・過度に攻撃的表現 | 任意 |
   | 違法行為 | 任意 |
   | 性的表現 | 任意 |
   | 子供に危害を加える表現 | 任意 |
   | 無断転載 | **必須**（転載元の情報を入力） |
   | なりすまし | **必須**（なりすまし元の情報を入力） |
   | その他 | **必須** |

2. **詳細入力フィールド**: テキスト入力。上表の「必須」理由を選択した場合は入力必須（空送信を弾く）。それ以外は任意

3. **「非表示にする」チェックボックス**: 対象コンテンツをまだ非表示にしていない場合のみ表示。**デフォルト ON**

4. **ボタン（縦並び）**:
   - 「通報する」: 通報を送信（理由未選択・必須詳細未入力時は無効）。送信後モーダルを閉じ、上部に「通報しました」フラッシュメッセージを表示。チェックボックスがONの場合は非表示処理も同時に実行し「通報しました」メッセージに「非表示にしました」を追記
   - 「キャンセル」: モーダルを閉じる（通報も非表示もしない）

### 6.6 通報スパム防止

通報システムの悪用を防ぐため、以下の3段階の制限を設ける。

| 制限 | 内容 |
|---|---|
| 通報者1人あたりの1日上限 | 100件（既存。超過時は 429 を返す） |
| 同一対象への通報頻度 | 同一ユーザー・ワールド・商品への通報は **1日1件**まで（超過時は「本日すでにこのコンテンツを通報済みです」を表示し送信ブロック） |
| スパム通報フラグ | 同一対象が **1時間以内に 20 件以上**の通報を受けた場合、管理画面の違反報告管理（セクション 8.4）に「集中通報フラグ」を付与。管理者が優先的に確認できるよう一覧の先頭に表示される |

- 集中通報フラグはあくまで管理者の確認優先度を上げるためのもの。自動的にユーザーへの制裁は行わない
- トラストシステム（セクション 21）は「重複なし通報者数」でカウントするため、同一ユーザーが繰り返し通報してもトラストポイントへの影響は1回分のみ

### 6.7 違反報告の保持期間

`user_violation_reports` テーブルのレコードは **7年間保持**する。

- 違反報告は `reporter_id` / `target_id` ともに `users.id` を参照するため、アカウント削除後も個人情報は切り離された状態でレコードが維持される
- 7年経過したレコードの物理削除バッチは**初期実装では省略**する（サービス開始から7年間は対象レコードが存在しないため不要）。将来必要になった時点で実装する

## 8. 管理画面（Web）

管理画面はGo `api/` サービスが `/admin` 以下で提供するWebインターフェースで、ログイン認証が必要。

### 8.0 管理者ロール

管理者アカウントには以下の 3 ロールを設ける。ロールは `admin_users` テーブルの `role` カラムで管理し、`/admin` ミドルウェアが各エンドポイントで検証する。

| ロール | 権限範囲 |
|---|---|
| `moderator` | アバター審査（8.2）・ユーザー管理（8.3）・違反報告管理（8.4）・ワールド管理（8.6）・操作ログ閲覧（8.8）のみ。財務データへのアクセス不可 |
| `admin` | `moderator` の全機能 ＋ ショップ管理（8.1）・トラストレベルロック操作 |
| `super_admin` | 全機能（`admin` の全機能 ＋ 売上管理 8.5・取引キャンセル管理 8.7・システムアラート 8.9・管理者アカウント管理） |

**権限マトリックス（エンドポイント × ロール）:**

| 機能 | moderator | admin | super_admin |
|---|---|---|---|
| アバター審査（承認・拒否） | ✓ | ✓ | ✓ |
| ユーザー管理（BAN・制限） | ✓ | ✓ | ✓ |
| 違反報告管理 | ✓ | ✓ | ✓ |
| ワールド管理（無効化・強制削除） | ✓ | ✓ | ✓ |
| 操作ログ閲覧 | ✓ | ✓ | ✓ |
| トラストレベルロック・解除 | ✗ | ✓ | ✓ |
| ショップ管理（商品登録・編集・削除） | ✗ | ✓ | ✓ |
| 売上管理・補正係数登録 | ✗ | ✗ | ✓ |
| 取引キャンセル管理（返金処理） | ✗ | ✗ | ✓ |
| システムアラート閲覧・操作 | ✗ | ✗ | ✓ |
| 管理者アカウント作成・ロール変更 | ✗ | ✗ | ✓ |

- `super_admin` のみが別の管理者アカウントを作成・ロール変更できる
- 全操作は `admin_audit_logs` に記録される（セクション 8.8・`docs/api-abstract.md` セクション 10）
- 権限外エンドポイントへのアクセスは 403 を返す

### 8.1 ショップ管理

| 機能 | 概要 |
|---|---|
| 商品登録 | アバター・アクセサリのGLBファイル・テクスチャ・価格・Edit OK/NGフラグを登録 |
| ワールドオブジェクト商品登録 | GLBファイル・テクスチャ・価格・コライダーサイズ・スケールロックフラグを登録。スケールロック ON の場合ユーザーはワールドエディタで拡大縮小不可（デフォルト: OFF） |
| ショップクリエイター登録 | クリエイター情報を登録し、ユーザーアカウントと紐づけ（任意） |
| 商品管理 | 商品の有効/無効切り替え・編集・削除 |

### 8.2 アバター審査画面

アップロードされたアバターを一覧表示し、審査を行う。

**フィルター（複数組み合わせ可）:**

| フィルター | 内容 |
|---|---|
| 要確認 | `avatars.needs_review = true`（新規ユーザー・通報済みユーザー・大容量テクスチャ。詳細: `docs/api-abstract.md` セクション8） |
| 通報あり | アバターに対して違反報告が 1 件以上ある |
| 新規ユーザー | アップロードしたユーザーの `trust_level` が `visitor` または `new_user` |
| 未審査 | `needs_review = true` かつ管理者による確認操作が未実施 |

デフォルト表示は「要確認」フィルターを適用した状態。

**デフォルト表示（常時）:**

- 前面・背面スクリーンショット（optimizer が生成）
- Diffuseテクスチャ画像
- アップロードユーザーの @name・trust_level・未処理違反報告件数

**3Dビューア（オンデマンド）:**

- 一覧の各エントリに「3Dで確認」ボタンを配置する
- ボタンクリック時のみ、エントリ内に Three.js + `@pixiv/three-vrm` による WebGL キャンバスをインライン展開する
- VRM ファイルは `@pixiv/three-vrm`（VRM 1.0対応）、GLB ファイルは Three.js 標準の `GLTFLoader` で読み込む
  - `@pixiv/three-vrm` は Three.js の `GLTFLoader` 拡張として動作するため、VRM・GLB 両対応を同一コードで実現できる
- 3Dビューア内の操作: マウスドラッグで回転・ホイールでズーム（`OrbitControls` を使用）
- ページ負荷を避けるため、Three.js / three-vrm のスクリプトはクリック時に動的インポート（lazy load）する
- 表示中に再度ボタンをクリックするとビューアを閉じる（トグル）

**操作:**

- 確認済みにマーク（`needs_review = false` に更新・問題なしと判断）
- BAN（BAN時は理由選択 + 警告種別選択）

### 8.4 違反報告管理画面

- 違反報告一覧（報告日時・報告者・対象ユーザー・種別・コメント）
- 対象ユーザーの違反報告一覧・警告履歴・BANログを表示
- 操作: 警告発行 / ユーザーBAN（警告のみ or 即時BANを選択可能）/ 対応完了マーク

**対応SLA:**

| 違反種別 | 目標対応時間 |
|---|---|
| 重大（脅迫・個人情報晒し・児童関連） | 24 時間以内に一次対応 |
| 著作権侵害（DMCA/著作権法に基づく申告） | 72 時間以内に一次対応（申告者への受付通知 + コンテンツ一時非公開） |
| 中程度（ハラスメント・スパム・不適切コンテンツ） | 7 日以内 |
| 軽微（その他） | 14 日以内 |

**著作権侵害の検知方針と拡張アーキテクチャ:**

現状（ローンチ時）は通報ベースのみで対応する。将来的に外部 AI モデレーション API（AWS Rekognition 等）を追加できるよう、VRM Optimizer パイプラインに `ContentModerationService` インターフェースを用意しておく。

```
// 現在の実装
type ContentModerationService interface {
    CheckImage(ctx context.Context, imageData []byte) (ModerationResult, error)
}

type NullModerationService struct{}  // 現状: no-op（チェックなし）

// 将来: ExternalModerationService が外部 API を呼び出す実装に差し替え可能
```

著作権申告フロー（DMCA 準拠）:
1. 申告者が管理画面または専用フォームから著作権侵害申告を送信
2. 対象コンテンツを即時「一時非公開」状態に変更（削除ではなくサスペンド）
3. 管理者が 72 時間以内に審査 → 侵害確認時はコンテンツ削除・ユーザー警告発行
4. 異議申し立て期間（14 日間）を経て最終処理

**対応完了後の通報者への通知:**

対応完了マークを付けた際に、通報者に対してアプリ内通知を送信する。通知内容は「ご報告いただいた内容を確認し、対応しました。」のみで、対応内容（BANしたかどうか等）は開示しない。

## 9. フレンド管理画面

### 9.1 フレンドシステム概要

フレンド関係は**相互承認型**とする。申請を受けた側が承認するとフレンド関係が成立する。

| 項目 | 通常ユーザー | プレミアム会員 |
|---|---|---|
| フレンド上限 | 100人 | 1000人 |

### 9.2 フレンド申請フロー

**ワールドモードから:**

1. アバターをタップ → タップUIの「フレンド申請」ボタン
2. 申請送信 → 相手の次回ログイン時または在線中にアプリ内通知で表示
3. 相手が承認 → フレンド成立

**フレンド管理画面から:**

1. 検索バーにユーザーID（`@name` 形式）を入力
2. 検索結果からユーザーを選択
3. 「フレンド申請」ボタンをタップ → 申請送信

### 9.3 フレンド管理画面の UI

アカウント表示モード（セクション 22.2）からもアクセスできる。

**タブ構成:**

| タブ | 内容 |
|---|---|
| フレンド | 承認済みフレンド一覧（アバターアイコン・ユーザー名・オンライン状態） |
| 申請中 | 自分が送った未承認の申請一覧（キャンセルボタン付き） |
| 申請受信 | 受信した未承認の申請一覧（承認 / 拒否ボタン付き） |

**フレンド一覧の操作:**

- 各フレンドをタップ → ユーザー情報パネル（2.4 のアバタータップUIと同様の内容）
- フレンド解除ボタン

### 9.4 フレンドのみルーム

- フレンドのみルーム: **ルーム作成者**のフレンドリストに存在するユーザーのみ参加可能
- 作成: ワールド詳細画面（セクション 2.10）の「フレンドのみのルームを作成」ボタン（設定パネルなし・ワンタップ作成）
- 参加: アプリ内通知招待・招待リンク・フレンドがいるルーム一覧画面（セクション 2.15）から行う。**参加時にサーバー側でフレンド関係を検証する**。非フレンドの場合は 403 `forbidden` を返す
- アプリ内通知招待（セクション 15）は、招待対象がルーム作成者のフレンドである場合のみ送信可能
- 人数上限はセクション 9.7 のルールに従いサーバーが自動設定（通常ユーザー: `min(6, world.max_players)`・プレミアム: `world.max_players`）

### 9.5 フォロワー限定ルーム

- フォロワー限定ルーム: **ルーム作成者をフォローしている**ユーザーのみ参加可能
- 作成: その他ルーム画面（セクション 2.14）の「フォロワー限定のルームを作成」ボタン
- 参加: アプリ内通知招待・招待リンクから行う。参加時にフォロー状態をサーバー側で検証する
- アプリ内通知招待（セクション 15）は、招待対象がルーム作成者をフォローしている場合のみ送信可能

### 9.6 招待制ルーム（プレミアム限定）

プレミアム会員のみが作成できるルーム種別。ルーム作成者が発行した招待リンク経由でのみ入室できる。

#### ルームの状態遷移

| 状態 | 説明 | 新規入室 |
|---|---|---|
| OPEN | 作成者がルームにいる | リンクが有効なら可 |
| LOCKED | 作成者が退室した（他の参加者は残留可） | 不可 |
| CLOSED | 全員退室 → ルームインスタンス消滅 | 不可（ルーム自体が存在しない） |

- 作成者が退室した時点で状態が LOCKED になり、それ以降は新規入室できない（作成者が再入室しても OPEN には戻らない）
- LOCKED 状態でも既存参加者はそのまま滞在でき、自由に退室可能
- 最後の1人が退室した時点でルームインスタンスが消滅し CLOSED 状態になる

#### 招待リンク仕様

| 項目 | 内容 |
|---|---|
| リンク形式 | `https://{app-domain}/invite/{token}`（iOS: Universal Links / Android: App Links でアプリへ誘導） |
| トークン | サーバー側で生成する予測不可能な乱数文字列（12文字英数字） |
| 有効期限 | 発行から **24時間** |
| 使用回数上限 | ルームの人数上限（`room.max_players`）と同じ回数（通常ユーザー作成: 最大 6 回・プレミアム作成: 最大 24 回） |
| 使用カウントの計算 | リンクを使って **初回入室したユニークユーザー数** をカウント。同一ユーザーの再入室はカウントしない |
| 再発行 | いつでも可能。再発行時に旧リンクは即時無効化（同時有効リンクは常に1本のみ） |

#### リンクの無効化条件

以下のいずれかに該当するリンクはサーバー側で参加拒否する:

1. 有効期限（24時間）を超過している
2. 使用回数（`room.max_players` の値）を超えている
3. ルームが LOCKED または CLOSED 状態
4. 再発行によって古いリンクになった

#### 招待リンク管理 UI（ルーム内）

ルーム内のメニューから操作可能（作成者のみ表示）:

| 操作 | 内容 |
|---|---|
| リンクをコピー | クリップボードにリンクをコピー（SNS・メッセージアプリで共有） |
| 再発行 | 新しいリンクを生成（旧リンクは即時無効。確認モーダルあり） |
| リンク情報表示 | 使用回数（X / maxPlayers の値）・有効期限（残り時間）を表示 |

#### リンクから参加する側のフロー

```
招待リンクをタップ
  ├── アプリ未インストール → ストアページへ誘導
  └── アプリ起動（またはフォアグラウンド復帰）
        └── サーバーでリンク検証
              ├── 有効 かつ ルーム OPEN → ルームへ参加
              └── 無効
                    ├── 有効期限切れ → "このリンクは有効期限が切れています"
                    ├── 使用回数上限 → "このリンクの上限に達しています"
                    ├── ルーム LOCKED/CLOSED → "ルームは現在入室を受け付けていません"
                    └── 無効なリンク（再発行済み等） → "招待リンクが無効です"
```

### 9.7 ルーム人数上限ルール

ルームの人数上限は以下の 2 層構造とする。

| 層 | 設定値 | 役割 |
|---|---|---|
| ワールド人数上限 | `world.max_players`（ワールド設定で指定・デフォルト 6） | ハードキャップ。ルーム人数上限の天井 |
| ルーム人数上限 | `room.max_players` | 実際の参加上限。ワールド上限以下で設定 |

**ルーム作成時の `room.max_players` 決定ルール（サーバー側で自動決定）:**

| 作成者プラン | room.max_players |
|---|---|
| 通常ユーザー | `min(6, world.max_players)` |
| プレミアム会員 | `world.max_players` |

- ルーム作成時にサーバー側で自動計算して設定する（作成者が個別に指定する UI は現状なし）
- 参加制御: 現在の参加人数 ≥ `room.max_players` の場合は入室拒否（409 Conflict）

### 9.8 ルームオーナーとオーナー移譲

各ルームには、ギミック状態同期のマスターとなる「ルームオーナー」が常に 1 人存在する。ここでいう「ルームオーナー」はギミック同期上の役割であり、招待制ルームのアクセス制御（セクション 9.6）とは独立した概念である。

**ルームオーナーの役割:**

- ギミックのステートマスターとして乱数生成・ギミック状態更新を担当する
- ルームの言語設定・人数上限設定の変更権は**ルーム作成者のみ**が保持する（オーナー移譲後の新オーナーは変更不可）

**オーナー移譲ルール:**

- ルームオーナーがネットワーク切断・退室・放置退室などで退席した場合、**入室時刻が最も古いユーザー（最長在室ユーザー）**が自動的に新オーナーになる
- 移譲はサーバー側で判定し、全クライアントに通知する

**元オーナーの再入室:**

- 元オーナーがルームに再入室した場合、通常の後から入室する場合と同じ方法で**現オーナーから状態同期**を受け取る
- オーナーは移譲したまま元のオーナーには戻らない

### 9.9 ギミック実行時の無限ループ検出

ルーム内でギミックが連鎖的に実行され、**同一フレーム内の連鎖数が 100 回**を超えた場合、`GimmickEngine` が無限ループと判定して以下の処理を行う。

**ルーム内全員に対して:**
- 画面上部にフラッシュメッセージを表示: 「ギミックエラーが発生したため、ルームをリセットします」
- ルーム内の全プレイヤーをスポーン地点へ移動（ルームリセット）
- ギミック状態をすべて初期状態にリセット
- ループを引き起こしたギミックルールの実行を以降のセッションでも停止する（そのルームのセッション中のみ。ワールドのデータは変更しない）

**ワールド作成者への追加通知:**
- 「ギミックに無限ループが検出されました。公開フロー（ワールド設定 → 新しいバージョンを公開）でループ箇所を確認・修正してください。」というアプリ内通知を送信する

---

**ブラシモード:**

- グリッドをタップまたはスライドすると、現在の高さに現在選択中の地形を配置する
- 既に地形が配置されている場合は**上書き**する

**消しゴムモード:**

- グリッドをタップまたはスライドすると、現在の高さにある地形を削除する

**図形モード:**

- グリッドをスライドした範囲に、選択した図形（四角形・円など、TBD）の形で地形を一括配置する

**タイプ変更モード:**

配置済みの地形をタップするたびに、そのブロックの形状を以下のサイクルで変更する:

```
[cube] → [ramp variants] → [diag variants] → [cube] → ...
```

**坂（ramp）に変更できる条件**: ブロックの真上が空いており、かつ隣接する側面の 1 方向以上が空いていること。条件を満たす向き（空いている側面の方向）が複数ある場合はタップのたびに向きが変わる。

**斜め（diag）に変更できる条件**: 隣接する側面のうち **2 方向が隣接していること**（直角に隣り合う 2 面が空いていること）。条件を満たす組み合わせが複数ある場合はタップのたびに向きが変わる。

坂・斜めをすべて一周したら立方体に戻る。

- タッチしたブロックが **ramp または diag の状態** で、かつ現在の条件をいずれの形状も満たさない場合: そのタップで立方体に戻す（自動復元はしない）
- 立方体の状態で条件を満たさない場合: そのブロックに対して変更できる形状がないため、エラー対象とする
- タッチした範囲に**変更できるブロックが 1 つもない**場合: フラッシュメッセージ「角になる地形をタッチしてください」を表示する

> 注: 一度タイプを変えた後に周囲の地形が変化して条件を満たさなくなっても、自動で立方体には戻らない。再度タイプ変更を行ったときに立方体に戻る。

**範囲選択モード:**

- グリッドをスライドした矩形範囲を選択する
- 選択範囲外のグリッドをタップすると選択が解除される
- 解除前に別の範囲をスライドすると**複数範囲**が選択される
- 選択中は**選択範囲内のブロックのみ**がブラシ・消しゴム・移動等の操作対象になる
- 移動後は移動後の範囲が新しい選択範囲になる
- 範囲選択ボタンを再タップすると選択方法を切り替えられる:

| 選択方法 | 動作 |
|---|---|
| 四角形 | スライドした矩形範囲を選択 |
| 塗りつぶし範囲 | タッチ地点から X 方向・Z 方向に、同じ地形種別（ramp・diag も同種とみなす）が隣接している限り選択を広げる。斜め隣接は接続しない |

範囲を選択した状態で**コピーボタン**が表示される。コピー後にペーストボタンが表示され、別の高さに移動してペーストすると選択範囲内の地形をその高さにコピー配置する。

**移動モード:**

- 選択中の範囲がある場合: その範囲内のブロックを XZ 平面方向に移動する
- 選択範囲がない場合: 現在の高さの地形全体を XZ 平面方向に移動する
- 拡大縮小は不可
- 配置可能範囲（63 × 63 グリッド）の外に出たブロックは削除する

#### 11.7.3 オブジェクトタブ

**タブ内固定ヘッダー（常時表示）:**

```
テクスチャコスト合計 / 4,096   オブジェクト数合計 / 400
```

**サブタブ:**

| サブタブ | 内容 |
|---|---|
| 利用中オブジェクト | ワールドに配置済みのオブジェクト・グループの一覧 |
| 保存・編集オブジェクト | セクション 11.3 の編集オブジェクトスロット + 保存バリアント |
| 所有オブジェクト | デフォルト・購入済みオブジェクト・特殊（スポーン位置・エリア・ポータル等） |

**利用中オブジェクトの表示:**

- サムネイルを**横方向に並べ**、行が溢れたら折り返す
- サムネイル左下に**そのオブジェクト種別のテクスチャコスト**を表示
- **ドラッグアンドドロップで並び替え**可能。並び順は Unity 上での描画順（レイヤー順）に対応する
- 複製時: 複製元の直後の位置に挿入する

**グループ:**

- 「グループ作成」ボタンを押すと新しいグループが作成される。選択中のオブジェクト・グループがある場合はそれらを含むグループを作成する
- グループは横一列分の行として表示され、グループ名の左側の **▶ / ▼ トグル**で内容の開閉ができる
- グループ内のアイテムは右にインデントして表示される
- グループ名をタップすると名前変更ダイアログが開く
  - 名前: 1〜20 文字（空文字不可）。デフォルト名は「グループ1」「グループ2」…の連番
- グループは**最大 4 段の階層構造**が可能。グループ作成上限は**合計 32 個**
- サムネイルをドラッグアンドドロップでグループ内に入れる / グループ外に出すことができる
- グループ選択時のギズモ操作: グループ内の**各オブジェクトにそれぞれ**移動 / 拡大縮小 / 回転を適用する（共通ピボットではなく個別適用）
  - いずれか 1 つでも適用不可（範囲外移動・最小サイズ超過等）の場合: 操作全体をキャンセルし、フラッシュメッセージ「移動できないオブジェクトがあったため移動に失敗しました」を表示

**複数選択:**

- **同一階層レベルのアイテムのみ**複数選択可能
- 階層が異なるアイテムを選択しようとした場合: フラッシュメッセージでエラーを表示
- 複数選択時は**移動のみ**可能（拡大縮小・回転は不可）

---

**オブジェクト操作（単体選択時）:**

- 利用中オブジェクト / グループをタップ → 選択状態になる
- 保存・編集 / 所有サブタブのオブジェクトをタップまたは 3D ビューにドラッグアンドドロップ → ワールド原点（0, 0, 0）に配置して選択状態になる
- 選択状態時に追加表示される要素:

| 要素 | 内容 |
|---|---|
| 複製ボタン | 選択中オブジェクトを全く同じ位置に複製（利用中一覧の直後に挿入） |
| 削除ボタン | 選択中オブジェクトをワールドから削除 |
| テクスチャ編集ボタン | オブジェクト編集画面（セクション 11.8）のテクスチャタブに遷移。編集結果はそのワールドにのみ適用（ワールドスコープカスタマイズ）。「保存バリアントに保存」ボタンで他ワールドでも再利用可能なバリアントとして保存可（仕様詳細: `docs/unity-game-abstract.md` セクション8・`docs/world-creation.md` セクション 3.7） |

- ギズモは上部中央ボタンで移動 / 拡大縮小 / 回転を切り替える
  - 移動: グリッドスナップ（0.5m 単位）
  - 回転: Y 軸 45° 単位
  - 拡大縮小: W / D / H を 0.25m 単位で増減（詳細: `docs/world-creation.md` セクション 3.5）。ショップオブジェクトにスケールロックが設定されている場合は操作不可（ロックアイコン表示）

---

**重複・配置ルール:**

- 通常オブジェクトは他のオブジェクト・地形と自由に重複配置できる
- スポーン位置・ポータルは他のオブジェクト・地形と重複した状態での配置が可能だが、重複している間はフラッシュメッセージで警告を表示し、**ワールドをプレイ・公開することができない**
- 3D ビュー左上の**地形透過ボタン**を押すと全地形を半透明にして表示する（オブジェクトの配置確認用）

---

**スポーン位置オブジェクト:**

- コライダーサイズ: **1m × 1.5m × 1m**（Y up）、削除・回転・拡大縮小は不可
- スポーン原点: コライダーの (0.5, 0, 0.5) の位置（XZ 中央・Y 最下部）。プレイヤーはここから出現する
- 1 ワールドにつき正確に 1 つ必要（配置なしはワールド公開不可）
- 他のオブジェクト・地形と重複した場合: 警告フラッシュメッセージ表示・プレイ/公開不可

---

**ポータルオブジェクト:**

ポータルはいずれも **1m × 1.5m × 1m** のデフォルトオブジェクト。コライダーが他オブジェクト・地形と重複した場合は警告表示・プレイ/公開不可。

| 種別 | 色 | 配置上限 |
|---|---|---|
| ワールドポータル | （サムネイル表示） | 4 つ |
| ルーム内入口ポータル | 白 | ルーム内ポータル合計 8 つ以内 |
| ルーム内出口ポータル | 黒 | ルーム内ポータル合計 8 つ以内 |

**ワールドポータル（セクション 11.7.3 詳細）:**

- 選択時: オブジェクトタブに「移動先ワールドを選択」ボタンが表示される
- 移動先の選択方法: ワールド一覧（公開済みのワールドのみ）から選択 / ワールド ID を直接入力
- ワールドが設定されると、ポータルの領域中央にそのワールドの**サムネイル画像を角丸長方形で空中に浮かせた**ような見た目になる

**ルーム内ポータル:**

- 入口ポータル（白）: 選択時に「転送先出口ポータルを選択」ボタンを表示。ワールド内の出口ポータル一覧から選択する
- 出口ポータル（黒）: 設定なし（転送先のみ受け取る）
- 公開条件: 配置されている全ての入口ポータルに出口ポータルが設定されていること

| 範囲乱数 | 最小値〜最大値を入力。ルームオーナーが乱数を生成して全プレイヤーに共有する |

#### 11.7.5 設定タブ（ワールド設定パネル）

- 下部タブ「設定」から開く

**テストプレイ**

- 「▶ テストプレイ」ボタン: オフライン・シングルプレイヤーでワールドをテストする
  - テストプレイ中は画面左上に「＜」ボタンを表示。タップで編集画面に戻る
  - テストプレイ中は画面上に「ステートデバッグ表示」トグルボタンを表示（詳細は後述）

**ステートデバッグ表示**

- テストプレイ中および通常ワールドモード（Unity デバッグビルド限定）で表示されるトグルボタン
- ON にすると全ワールドステート（ID・名前・現在値）をオーバーレイ表示する
- テストプレイ中は設定タブのボタンでも、通常ワールドモードのデバッグビルドでは画面上のボタンで切り替える

**サムネイル**

- タップするとサムネイル設定モーダルを開く
- **写真から選択**: 端末ギャラリーから画像を選択 → 9:16 クロップ調整画面 → 確定でアップロード
  - クライアント側で解像度を正規化してからアップロード（サーバー側でも検証）
- **ゲーム内で撮影**: テストプレイモードに移行 → 撮影機能（セクション 2.7）でキャプチャ → 撮影後に確認モーダルを表示（「サムネイルに設定」/ 「撮り直す」）
  - 確認モーダルで「サムネイルに設定」を選択すると編集画面に戻りサムネイルが更新される

**ワールド基本設定**

- ワールド名の入力
- タグ入力（最大 5 個、1 タグ最大 20 文字）
  - テキストフィールドに入力して追加。追加済みタグはチップ形式で表示し × で削除可能
  - 上限到達時は追加フィールドを非表示
- 背景設定（単色 / グラデーション / テクスチャアップロード）
- 環境カラー設定: ワールド全体の雰囲気カラー。カラーピッカーで選択（デフォルト: 白）。HSV の V 値 0.25 未満は選択不可（詳細: `docs/world-creation.md` セクション 15.17 参照）
- 環境音設定（詳細: `docs/world-creation.md` セクション 14）
  - 内蔵サウンドライブラリから選択（「なし」含む）
  - 音量スライダー（0〜100%）: ユーザーの「ワールド効果音」設定に対する相対値
  - フェード時間は固定値（ユーザーによる変更不可）
- 人数上限: ワールド内で作成されるルームの人数上限ハードキャップ。デフォルト 6 人。通常ユーザーは 2〜6 人、プレミアム会員は 2〜24 人の範囲で設定可能。各ルームの実際の人数上限（`room.max_players`）はこの値以下で決定される（セクション 9.7 参照）
- プレイヤー高さによる非表示: トグルスイッチ（デフォルト ON）。ON のとき、プレイヤーの真上にレイを飛ばして当たった地形以上の高さの地形・オブジェクト・他プレイヤーを非表示にする（詳細: `docs/world-creation.md` セクション 15.11 参照）
- 公開/非公開の切り替え

**バージョン情報**

- 公開バージョンの番号を表示する（例: `バージョン 3`）
- バージョン番号はサーバーが **1 から始まる整数**で自動採番し、公開するたびにインクリメントされる（ユーザーは変更不可）
- 未公開（初回公開前）の場合は「未公開」と表示する

**公開・バックアップ**

- **「新しいバージョンを公開」ボタン**: 現在のドラフトを公開バージョンとして確定する
  - 詳細は 11.7.6「公開フロー」参照
- **「バックアップを保存」ボタン**（プレミアム会員のみ表示）: 最大 5 枠のバックアップスロット
  - バックアップに保存される内容: ワールド設定・地形データ・オブジェクト配置（オブジェクト本体のテクスチャ等コンテンツは含まない）
  - スロット選択後に保存。既存スロットに上書きする場合は確認ダイアログを表示
  - バリデーション: 異常なファイルサイズ（上限超過）のみチェック。コライダー検証は行わない
  - スロット一覧には保存日時を表示
- **「バックアップを復元」ボタン**（プレミアム会員のみ表示）: 保存済みバックアップからドラフトを復元する
  - スロット選択後に確認ダイアログを表示。復元するとドラフトの現在内容は上書きされる

#### 11.7.6 公開フロー

```
「新しいバージョンを公開」ボタン → バリデーション
  ├── ワールド名未入力 → エラー表示
  ├── サムネイル未設定 → エラー表示
  ├── スポーン位置未設定 → エラー表示
  ├── 入口ポータルに出口未設定 → エラー表示
  ├── オブジェクト重複あり → エラー表示
  ├── ファイルサイズ超過 → エラー表示
  ├── テクスチャコスト上限超過 → エラー表示
  ├── バージョン番号が INT 上限（2,147,483,647）に達している → エラー表示（「バージョン番号が上限に達しているため公開できません」）
  ├── ギミックループ検出（内部テストプレイ実行） → エラーパネルを表示（詳細下記）・公開をブロック
  └── すべて OK → 公開バージョンとしてサーバーへ送信・バージョン番号をインクリメント・公開完了

**ギミックループ検出エラーパネルの表示内容:**

ループを引き起こしているルール名を列挙し、ユーザーが修正箇所を特定できるようにする。

```
【ギミックに無限ループが検出されました】

以下のルールが無限ループを引き起こしています。
ギミック設定を確認して修正してください。

  ・ルール「扉を開く」
  ・ルール「ライトを切り替える」

（公開するには上記ルールを修正してください）
```

- 「ギミック設定へ」ボタンでギミックエディタへ直接遷移する
- ループに関与するルールをギミックエディタ内でハイライト表示する（可能な場合）
- エラーパネルを閉じても公開は実行されない

※ バージョン更新が完了した時点で、そのワールドの既存ルームは「旧バージョン」となり新規入室不可になる（既在室者はルーム終了まで継続利用可）

※ドラフト保存（自動保存）は上記バリデーションなしで随時実行される
  ただしスポーン位置未設定 or 入口ポータルに出口未設定 の場合は保存時に警告を表示する（保存は実行される）

#### 11.7.7 ドラフト保存の競合検出（楽観的ロック）

同一ユーザーが複数デバイスで同じワールドを同時編集した場合の上書き防止のため、楽観的ロックを使用する。

- ドラフト保存リクエストには**現在のドラフトバージョン番号**（`draft_version`）を含める
- サーバーはリクエストの `draft_version` と DB のバージョンを比較し、一致する場合のみ保存してバージョンをインクリメントする
- 不一致の場合（別デバイスが先に保存済み）は **409 Conflict**（`reason: "draft_version_conflict"`）を返す

**Unity クライアント側の競合時挙動:**

```
409 draft_version_conflict を受信
  → フラッシュメッセージ: 「別のデバイスで編集が保存されました。最新データを取得しますか？」
  → ［取得する］: サーバーから最新ドラフトを再取得（未保存の変更は破棄される）
  → ［キャンセル］: 現在の編集を継続（次の自動保存時に再度競合する可能性がある）
```

- 409 が続く場合でも自動保存はリトライせず、ユーザーの明示的な選択を待つ
```

## 14. 非表示機能

### 14.1 仕様

- 非表示は**一方向**（相手には通知されない）
- 非表示数の上限なし
- レート制限: 1日50件

**非表示後の効果（非表示にした側から見た挙動）:**

| 項目 | 効果 |
|---|---|
| アバター描画 | 非表示ユーザーのアバターを不可視にする（クライアント側でレンダリングをスキップ） |
| 音声 | Vivox の当該参加者をミュート |
| フレンド関係 | 即時解除（サーバー側） |
| フレンド申請 | 非表示中ユーザーへの申請ボタンを非表示。非表示ユーザーからの申請はサーバー側でサイレント破棄（申請者には **201 Created を返す**が、実際には申請レコードを作成しない。プライバシー保護のため非表示設定を申請者に知らせない） |
| フォロー状態 | 変更しない（フォローは維持されるが、必要であれば別途解除可能） |

**内部処理:**

ギミックやルーム状態の整合性を保つため、非表示ユーザーのアバターはサーバー・他クライアント上では通常通り存在する。非表示はあくまで「非表示にした側のクライアント」での描画・音声のみに影響する。プレイヤー数カウント等のゲームロジックには影響しない。

### 14.2 非表示リスト管理画面

ユーザー設定画面から遷移。

- 非表示ユーザーの一覧（アバターアイコン・ユーザー名）
- 各エントリに「非表示解除」ボタン
- 解除後: 次にルームに入り直した時点で再表示・再音声有効化

### 14.3 ユーザー非表示の導線

| 導線 | 操作 |
|---|---|
| アバタータップUI（ワールドモード内） | 「非表示」ボタン → 確認モーダル → 実行 |
| インワールドメニュー ルームタブ メンバーパネル | その他アクション（…）→「非表示にする」→ 確認モーダル（セクション 2.11.2） |
| 通報モーダル | 「非表示にする」チェックボックスが ON の状態で通報を送信（セクション 6.5） |

## 15. 通知システム

### 15.1 アプリ内通知

アプリ起動中（フォアグラウンド）に受信できる通知。

**通知種別:**

| 種別 | 内容 | プッシュ通知対象 |
|---|---|---|
| フレンド申請受信 | @name からフレンド申請が届きました | なし |
| ワールド公開通知 | フォロー中の @name がワールドを公開しました | あり（ユーザーが有効化時） |
| 商品販売開始通知 | フォロー中の @name の商品が販売開始しました | あり（ユーザーが有効化時） |
| コイン失効通知（30日前） | お持ちのコインが 30 日後に一部失効します | あり（ユーザーが有効化時） |
| コイン失効通知（7日前） | お持ちのコインが 7 日後に一部失効します | あり（ユーザーが有効化時） |

- 通知センター（ベルアイコン）からアプリ内通知一覧を確認できる
- 既読・未読管理（未読件数バッジ）
- 通知タップで対象画面へ遷移（ワールド詳細 / ショップ商品詳細 / フレンド管理画面）

### 15.2 プッシュ通知

バックグラウンド・スリープ中でも受信できる通知（対象種別は上表参照）。

**デフォルト動作:**

- 初期状態: プッシュ通知 OFF（アプリ内通知のみ）
- ユーザーが設定画面の各種別ベルマーク（🔔）を ON にすると有効化
- iOS: `UNUserNotificationCenter` による通知権限リクエスト（ベルマークを初めて ON にした時にリクエスト）
- Android: `POST_NOTIFICATIONS` 権限リクエスト（同上）

### 15.3 レート制限

| 制限対象 | ルール |
|---|---|
| 同一フォロー元ユーザーからの通知 | 受信者1人に対して1日1件まで（ワールド公開・商品販売開始を合算） |
| 全体プッシュ通知受信数 | 1ユーザーが1日に受け取るプッシュ通知の総数: **1日12件** |

- 上限超過時: プッシュ通知は送信せず、アプリ内通知のみ積み上げる
- 同日中に同一フォロー元から複数イベントが発生しても、通知は最初の1件のみ送信

### 15.4 通知設定 UI

ユーザー設定画面内の「通知設定」セクションから操作。

| 設定項目 | ON/OFF | ベルマーク（プッシュ） |
|---|---|---|
| フレンド申請通知 | ○ | — |
| フォロー: ワールド公開通知 | ○ | ○ |
| フォロー: 商品販売開始通知 | ○ | ○ |
| コイン失効通知 | ○ | ○ |

- ON/OFF: アプリ内通知・プッシュ通知をまとめて無効化
- ベルマーク: プッシュ通知のみを個別に有効化（ON/OFF が OFF の場合はグレーアウト）
- プッシュ通知を有効化した場合、OS の通知権限が未付与であれば権限リクエストダイアログを表示

## 16. いいね機能

### 16.1 概要

- ユーザーはワールドおよびショップ商品（アバター・アクセサリ・ワールドオブジェクト）に「いいね」を付けられる
- いいね数は公開される（ワールド一覧・ワールド詳細・商品一覧・商品詳細に表示）
- 自分のいいねは後から取り消し可能
- 自分のワールド・自分の商品にはいいね不可（サーバーで拒否・403）
- いいねを受けた際のアプリ内通知・プッシュ通知は送信しない

### 16.2 操作仕様

| 操作 | 挙動 |
|---|---|
| いいね | 対象にいいねを追加。同一対象への重複いいねはサーバーで拒否（409） |
| いいね解除 | 自分のいいねを取り消す |
| 自己いいね | サーバーで拒否（403）。自分のコンテンツにはいいねボタンを非表示 |

- レート制限: 1日300件

### 16.3 いいね順ソート

ワールド一覧・ショップ商品一覧でいいね数の多い順に並べ替えられる。

**実装方針（クエリ性能）:**

- `worlds` / `products` テーブルにデノーマライズされた `likes_count INTEGER NOT NULL DEFAULT 0` カラムを持つ
- いいね時にインクリメント・いいね解除時にデクリメント（アプリケーションレベルで更新）
- `likes_count` カラムに降順インデックスを作成し、ソートクエリの集計コストを O(1) に抑える
- `world_likes(world_id, user_id)` / `product_likes(product_id, user_id)` テーブルに UNIQUE 制約を設け、重複防止と参照整合性を保証する

## 18. アカウント削除

### 18.1 アクセス方法

ユーザー設定画面（Phase 10）の末尾に「アカウントを削除する」リンクを設置する（破壊的操作であるため赤字テキストリンク推奨）。

### 18.2 削除フロー

```
「アカウントを削除する」をタップ
  └── 確認ダイアログを表示
        ├── [キャンセル] → 閉じる
        └── [削除する] → API へ削除申請を送信
              └── 即時ログアウト → ログイン画面へ遷移
```

**確認ダイアログの文言:**

```
アカウントを削除しますか？

・アカウントは即時ログアウトされ、アクセスできなくなります
・表示名・@name・連携サービス情報は即時削除されます
・アップロードしたアバター・ワールド・オブジェクトなどのデータは
　30日後に完全に削除されます
・購入履歴・コイン取引履歴は削除されません
・削除したアカウントは復旧できません

[削除する]　[キャンセル]
```

### 18.3 削除タイムライン

| タイミング | 処理内容 |
|---|---|
| 削除申請時（即時） | `active_users.deleted_at` を記録（ソフトデリート） |
| 削除申請時（即時） | 全セッショントークンを無効化（ログアウト） |
| 削除申請時（即時） | 表示名・@name・メールアドレス・ソーシャルプロバイダー紐付けを即時 NULL 化（GDPR 対応） |
| 削除申請時（即時） | 公開中のワールドをすべて非公開へ変更 |
| 削除申請時（即時） | ログイン・API アクセスを拒否（`active_users` レコードが削除済みとして扱われるため 403） |
| 30 日後（バッチ処理） | `active_users` レコードを物理削除（残存する設定・信頼スコアが消去される） |
| 30 日後（バッチ処理） | アップロードリソースを物理削除（VRM・GLB・テクスチャ・レイヤー画像・ワールド定義 JSON・アトラス） |
| 30 日後（バッチ処理） | @name を解放（他ユーザーが取得可能に） |

**削除後も保持するデータ（法的・監査目的）:**

- コイン購入履歴（`platform_transaction_id` 含む・`users.id` への参照は維持）
- コイン消費履歴（`users.id` への参照は維持・個人情報は `active_users` 削除により消去済みのため匿名化は不要）
- 取引キャンセル記録
- 違反報告受信履歴（`users.id` への参照は維持）

### 18.4 管理画面からのアクセス（セクション 8.3 参照）

- 削除申請済みアカウントは管理画面で「削除済み」として参照できる（PII は即時 NULL 化済みのため個人情報は表示されない）
- アカウントの復元機能は提供しない（ユーザー向け・管理者向けともに不可）

## 21. テクスチャ編集UI仕様

アバター編集画面（セクション 3.5.1）・アクセサリ編集画面（セクション 3.6.1）・オブジェクト編集画面（セクション 11.8.1）・ワールドエディタ内テクスチャ編集に共通するUIの詳細仕様。実装詳細は `docs/unity-game-abstract.md` セクション8参照。

### 21.1 テクスチャモード

テクスチャ編集には 2 つのモードがある。

| モード | 概要 |
|---|---|
| 詳細テクスチャモード | フルペイント機能。ブラシ・レイヤー・範囲選択等が使える。デフォルト |
| 簡単テクスチャモード | プリセットから選択するシンプルモード（詳細仕様別途定義） |

詳細→簡単への切り替えは「その他」メニュー →「簡単テクスチャを使う」から行う（セクション 21.10 参照）。

### 21.2 テクスチャ編集バー（詳細テクスチャモード）

テクスチャ編集タブ下部に配置されるツールバー。

| ボタン | 機能 |
|---|---|
| 色選択 | 色選択パネルを開く（セクション 21.3） |
| ブラシ | ブラシモードに入る（セクション 21.4） |
| 消しゴム | 消しゴムモードに入る |
| 塗りつぶし | 塗りつぶしモードに入る（セクション 21.5） |
| 図形 | 図形モードに入る（セクション 21.6） |
| 範囲選択 | 範囲選択モードに入る（セクション 21.7） |
| 移動・拡大・縮小 | 移動拡大縮小モードに入る（セクション 21.8） |
| レイヤー | レイヤーパネルを開く（セクション 21.9） |
| その他 | 追加メニューを開く（セクション 21.10） |

### 21.3 色選択パネル

| 要素 | 内容 |
|---|---|
| 色相カラーサークル | 外周の円で色相（Hue）を選択。選択位置に白枠の円マーカーが表示される |
| 明度・彩度四角形 | カラーサークル内側の四角形。横軸=彩度・縦軸=明度（現在の色相における色が表示される）。選択位置にマーカーを表示（明度 < 128/255 → 白枠、明度 ≥ 128 → 黒枠） |
| 透明度バー | 横向きバーで透明度（α）を指定 |
| RGBA数値入力 | R / G / B / A を数値で直接入力 |
| 色利用履歴 | 実際に使用した色を最大 16 種類保持。タップするとその色を選択できる |

### 21.4 ブラシモード

ブラシボタンを押すとブラシモードに入る。

- 画面左に**サイズ調整バー**が表示され、ブラシサイズを 1〜255（デフォルト: 1px）に変更できる
- ブラシモード中にブラシボタンを再度押すとブラシ種類を選択できる

| ブラシ種類 | 動作 | デフォルト |
|---|---|---|
| 円形（アンチエイリアスなし） | 指定色でくっきり塗る。ドット絵・ピクセルアートスタイルが可能 | ○ |
| 円形（アンチエイリアスあり） | 指定色を周囲ピクセルと混合しながら塗る（ソフトエッジ） | |

### 21.5 塗りつぶしモード

塗りつぶしボタンを押すとモードに入る。タップした位置と同じ色の連続した区画を塗りつぶす。

- 塗りつぶしモード中にボタンを再度押すと設定パネルが開く

| 設定項目 | 内容 |
|---|---|
| 他レイヤーを参照する / しない | ON にすると他のレイヤーの色も考慮して塗りつぶし範囲を判定する |
| 許容値 | 同じ色と見なす色差の上限を設定する（低→厳格・高→広範囲） |

### 21.6 図形モード

図形ボタンを押すとモードに入る。スライドした範囲に図形を描画する。

- 図形モード中に図形ボタンを再度押すと描画する図形を選択できる

| 図形 | 描画方法 |
|---|---|
| 四角形 | スライドした矩形範囲に描画 |
| 円 | スライドした矩形範囲に内接する円を描画 |
| 直線 | スライドした始点〜終点に直線を描画 |

### 21.7 範囲選択モード

範囲選択ボタンを押すとモードに入る。スライドした範囲を選択する。

- テクスチャ外側をタップすると選択解除
- 選択解除前に別の範囲を選択すると複数範囲選択になる
- 選択中はその範囲のピクセルのみが編集対象（ブラシ・消しゴム・図形・移動拡大縮小等）
- 移動または拡大縮小した場合、選択範囲は操作後の範囲に更新される
- 範囲選択直後に下部に**コピーボタン**が表示される。押すと**ペーストボタン**に変わる。別レイヤーに移動してペーストを押すと、コピー元レイヤーの範囲内の画像がそのレイヤーにペーストされる
- 範囲選択モード中にボタンを再度押すと選択方法を変更できる

| 選択方法 | 内容 |
|---|---|
| 四角形 | スライドした矩形範囲を選択 |
| 円 | スライドした矩形に内接する楕円範囲を選択 |
| 塗りつぶし範囲 | タップ位置に塗りつぶし判定が適用される範囲を選択 |

### 21.8 移動・拡大縮小モード

移動・拡大縮小ボタンを押すとモードに入る。

- スライドで現在選択中のレイヤーを移動する
- ピンチイン・アウトで現在選択中のレイヤーを拡大縮小する（通常はテクスチャビューのズームになるが、このモードではレイヤー操作になる）
- モード中にボタンを再度押すと補間方法を選択できる

| 補間方法 | 内容 |
|---|---|
| カクカク | Nearest Neighbor 法（ドット絵・ピクセルアート向け） |
| なめらか | Bicubic 法（写真・イラスト向け） |

### 21.9 レイヤーパネル

レイヤーボタンを押すとパネルが開く。

#### パネル内ボタン

| ボタン | 機能 |
|---|---|
| 新規レイヤー追加 | 通常ラスターレイヤーを追加 |
| 新規効果レイヤー追加 | 色調補正レイヤーを追加（1テクスチャにつき1枚まで・通常レイヤーの枚数制限とは別枠） |
| レイヤー取り込み | ファイルから画像を読み込んで新規レイヤーとして追加。サイズ不一致時は「解像度が異なるため自動でサイズ調整されます」モーダル表示後に Bicubic 法でリサイズ |
| レイヤーグループ作成 | レイヤーをフォルダ状のグループにまとめる |

#### レイヤー一覧

- ドラッグアンドドロップで上下に並び替え・グループへの出し入れが可能
- タップでそのレイヤーを選択状態にする
- レイヤーグループをタップするとグループ内の全レイヤーをまとめて選択する

各レイヤーの表示要素:

| 要素 | 内容 |
|---|---|
| サムネイル | レイヤー内容のプレビュー |
| レイヤー名 | テキスト表示 |
| 表示・非表示ボタン | レイヤーの表示を切り替える |
| その他アクションボタン | 削除 / 画像書き出し / 透明度調整 / 下のレイヤーにマスク / レイヤーロック |

#### 特殊レイヤー（レイヤー枚数制限にカウントされない）

**UV レイヤー（レイヤー一覧の最上段に固定）:**

- UV レイアウトの枠線を表示するオーバーレイレイヤー
- 表示・非表示の切り替えのみ可能（通常のレイヤーとして編集不可）
- 枠線色のデフォルトは `#80808080`。レイヤー右側の色部分をタップすると色選択パネル（セクション 21.3 と同様）で変更可能

**ベースレイヤー（レイヤー一覧の最下段に固定）:**

- 不透明 / 透明を 2 値（0/1）で管理するマスクレイヤー
- ブラシで描くと色にかかわらず不透明（α=255）になる。消しゴムで透明（α=0）になる。半透明は不可
- テクスチャの最終出力は「すべての通常レイヤーにこのベースレイヤーをマスクした結果」になる（出力に半透明ピクセルは発生しない）
- デフォルト: 全面不透明
- 非表示にすると通常描画レイヤーはすべて非表示になる（UV レイヤーは除く）

### 21.10 その他メニュー

| 項目 | 内容 |
|---|---|
| 画像として書き出し | ベースレイヤー・効果レイヤーも含め現在表示中のすべてのレイヤーを統合してファイルに保存 |
| テクスチャサイズ変更 | テクスチャ全体のサイズを変更してコストを調整。補間方法（カクカク / なめらか）を選択 |
| 簡単テクスチャを使う | 確認モーダル（「カスタムテクスチャの内容は削除されますがよろしいですか？」/ 「削除する」・「キャンセル」）→「削除する」で簡単テクスチャモードに切り替わり、デフォルトテクスチャ「ブロック」が適用される |

### 21.11 2D キャンバス操作

- ピンチイン・アウトでキャンバスの表示拡大縮小（最大 ×8 程度・最小は全体表示）
- 移動・拡大縮小モード（セクション 21.8）以外でのピンチ操作はキャンバスビューのズーム

### 21.12 テクスチャ編集終了時の処理

テクスチャタブを離れる際（他タブへの切り替え・画面からの離脱いずれも）に以下の処理が自動実行される:

1. テクスチャ範囲外にはみ出したデータを削除
2. 全レイヤーを統合した統合画像を生成
3. レイヤー構造とレイヤー画像（変更分のみ）を保存データとして確定

## 22. アカウント表示モード

右上の自分のアイコン（セクション 1.1）をタップすると遷移する。自分のプロフィール・アカウント情報の確認と編集、フレンド・フォロー関係の管理を行う画面。

### 22.1 プロフィール・アカウント情報

| @name | 現在の @name を表示。変更ルールはセクション 5.1 参照（プレミアムのみ・90日制限） |

### 22.2 フレンド管理画面

セクション 9.3 のフレンド管理画面と同一内容（アカウント表示モードからのアクセス経路を追加）。

**フレンド一覧（承認済みフレンドタブ）から他ユーザーを選択した場合:**

- そのユーザーのアカウント情報パネルを表示する
- パネルには以下のボタンを表示する:

### 22.3 フォロー管理画面

セクション 13.2 のフォロー管理画面と同一内容（アカウント表示モードからのアクセス経路を追加）。

**フォロー中タブ・フォロワータブからユーザーを選択した場合:**

- そのユーザーのアカウント情報パネルを表示する
